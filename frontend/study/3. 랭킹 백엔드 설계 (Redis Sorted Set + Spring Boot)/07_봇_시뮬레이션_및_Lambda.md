# 섹션 7: 봇 시뮬레이션 및 Lambda 부하 테스트

DayPoo 랭킹 시스템이 실제로 훌륭하게 작동하는지 개발 환경에서 증명하려면 수천 명의 유저가 끊임없이 경쟁하는 데이터 모델이 필요합니다. 이를 위해 **봇(Bot) 기반의 시나리오**가 운영됩니다.

## 1. 시나리오 봇 시스템 개요 (`simulation` / `bot` 폴더 구조)
백엔드는 실제 사람과 유사하게 움직이는 봇 객체를 가집니다.
1. **Seeder:** 초기 가입 봇 100~500명 생성 및 화장실 데이터 확보
2. **Scenario:** 
   - 봇은 무작위 확률로 지역 내 특정 화장실을 배회합니다.
   - 일부는 화장실 배변 기록 이벤트(`PooRecordCreatedEvent`)를 펑펑 쏴댑니다.
   - 일부 봇은 건강 정보 리포트를 발행하여 건강왕 집계 서버에 트래픽 핑을 줍니다.

## 2. Terraform AWS Lambda 인프라 확장
단일 Spring Boot 컨테이너에서 억지로 봇을 돌리면 어플리케이션 자체 성능 스펙이 왜곡됩니다. 따라서 외부에서 무작위성 HTTP REST 요청을 쏘는 형태로 부하 발생기를 분리합니다.

- **AWS EventBridge (CloudWatch Events):** N분마다 주기적 시그널 발생.
- **AWS Lambda (Python/Node):** 시그널을 받으면 임의의 위치, 임의의 봇 Token을 가지고 DayPoo 백엔드 API를 융단폭격 (기록 생성 500건).
- **효과:** 순수 백엔드 서버(Java)는 실제 클라이언트 트래픽이 몰릴 때 Redis ZSet와 EventListener가 제대로 병목 없이 해소되는지 메트릭 보드를 통해 모니터링 할 수 있습니다.

---

> [!NOTE] 랭킹 설계 핵심
> **Stress Test 와 아키텍처 검증**
> Redis Sorted Set 연산 속도는 O(log N) 입니다. N이 천만 단위로 갈 때 레이턴시가 50ms 미만으로 유지되는가? 데이터베이스 락(Lock) 충돌 현상은 없는가? 
이 모든 것들은 탁상공론이 아닌 **자동화된 시뮬레이터 봇**을 통해서만 증명할 수 있습니다. Lambda를 통한 서버리스 아키텍처를 도입해 부하 발생 비용을 최적화시킨 것이 백엔드 인프라 설계의 방점입니다.
