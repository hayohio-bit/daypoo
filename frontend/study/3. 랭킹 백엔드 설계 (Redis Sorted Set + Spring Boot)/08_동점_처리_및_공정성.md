# 섹션 8: 동점 처리 및 공정성 방파제

온라인 랭킹 시스템에서 "동점자가 수만 명일 때 어떻게 공정하게 서열을 가르는가?" 그리고 "오토 클릭 프로그램으로 점수를 복사하는 어뷰저들을 어떻게 막는가?"는 운영의 핵심입니다.

## 1. Redis ZSet의 기본 동점자 정렬 기준 (Lexicographical Order)
점수가 백분위 단위까지 똑같은 A 유저와 B 유저가 있을 때, Redis Sorted Set (`ZADD`)은 기본적으로 유저 **Value(문자열)의 사전표기순(알파벳순)**으로 정렬을 수행합니다.

- 점수 `40.5`: 사용자 "Alice"
- 점수 `40.5`: 사용자 "Bob"
- 점수 `40.5`: 사용자 "Charlie"
- 이 경우 `ZREVRANGE` 결과로 Alice -> Bob -> Charlie 순으로 랭크됩니다.

**문제점:** 알파벳이 앞순서인 ID/특수문자를 가진 사용자가 무조건 유리해지는 결함이 있습니다!

### [심화 이론] 타임스탬프 인코딩 활용법 (해결책)
점수를 Long이나 Double로 저장할 때 자릿수 트릭을 이용하여 소수점 아래에 "달성한 시간(Unix Time의 역수)"을 은닉시키는 기법입니다.
- 기본 점수: 45
- 달성 시간: 2026-04-07 10:00:00 (초로 변환하면 값이 커짐. 제일 늦게 달성한 사람 점수를 깎아야 함)
- **제출 점수 공식:** `순수점수 + (1.0 - (현재시간 / 최대시간))`
- 이렇게 ZADD 시키면 완전한 동점자들 중에서 **먼저 도달한 사람이 더 높은 총점**을 가지게 되며 수식만으로 선착순 동점 해소가 가능합니다.

## 2. 시스템 레벨 점수 조작 방어 기법 (공정성)
현재 DayPoo API 서버는 악의적인 봇과 자동화 공격을 방어하기 위해 다음과 같은 진지가 구축되어 있습니다.

1. **AOP 기반 Rate Limiting 기법:**
   - 동일 IP 및 JWT 컨텍스트 유저가 1초 내에 기록 추가 요청을 N번 연사하면 AOP 필터에서 **429 Too Many Requests** 를 반환시키고 튕겨냅니다.
2. **애플리케이션 검증 로직:**
   - 사용자가 남긴 좌표의 **최소 이동 거리 및 소요 시간**을 백엔드에서 재검증하여, KTX도 불가능한 시간(2초 만에 서울화장실 -> 부산화장실 핑)에 좌표가 이동하면 점수 인정을 기각하거나 어뷰징 플래그를 설정합니다.

> **신뢰할 수 없는 사용자의 입력 (Zero Trust)**
> 모든 게임과 소셜 랭킹 서비스의 1원칙은 "클라이언트에서 보내온 점수는 거짓이다" 라는 것입니다. 프론트엔드가 보낸 완료 액션을 철저히 의심하고 서버 측에서 순수 배변 기록 좌표의 이격 거리, 그리고 동률 시 타임스탬프 기반 자동 감시망을 통해 깨끗한 랭킹 시스템을 운영해야 유저들의 신뢰를 얻을 수 있습니다.
