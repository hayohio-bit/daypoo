# 섹션 5: 랭킹 로테이션 및 스케줄러

DayPoo 시스템은 현재 시간 감쇠(Decay)와 같은 복잡한 실시간 뺄셈 공식보다는 주기적으로 랭킹 데이터를 통째로 **리빌트(Rebuild)** 하는 방식으로 데이터의 무결성과 회전율을 조절하고 있습니다.

## 1. @Scheduled를 통한 크론(Cron) 작업 실행

`RankingService` 내부에는 매일 새벽 4시(사용자 트래픽이 가장 적은 시간)에 동작하는 스케줄러가 탑재되어 있습니다.

```java
  @Scheduled(cron = "0 0 4 * * *")
  public void scheduledRankingRebuild() {
    log.info("[Ranking] 스케줄 랭킹 재구축 수행 (매일 04:00)");
    rebuildAllRankings();
  }
```

## 2. 분산 서버 환경에서의 Lock (분산 락)
랭킹 서버가 여러 대(Scale-Out) 떠 있을 경우, 새벽 4시에 3대의 서버가 동시에 `rebuildAllRankings()` 를 호출하면 DB에 막대한 다중 쿼리 부하가 발생하게 됩니다. 이를 해결하기 위해 Redis의 훌륭한 기능인 **원자적 락 기능(setIfAbsent)** 을 적용했습니다.

```java
  public void rebuildAllRankings() {
    String lockKey = "daypoo:lock:rebuild_rankings";
    // 1. Redis에 해당 Key가 없다면 즉시 생성(락 획득), 있다면 실패(락 획득 실패)
    Boolean acquired =
        redisTemplate.opsForValue().setIfAbsent(lockKey, "locked", java.time.Duration.ofMinutes(5));
        
    if (acquired == null || !acquired) {
      log.info("[Ranking] 이미 다른 프로세스에서 랭킹 재구축 중입니다. 스킵합니다.");
      return; // 2. 락 획득 실패한 서버들은 즉시 메서드를 종료함
    }

    try {
        // 3. 락을 획득한 1대의 서버만이 랭킹 재구축 로직을 독점 수행
        log.info("[Ranking] 전체 랭킹 재구축 시작 (Atomically)...");
        // ... (Rebuild 로직) ...
    } finally {
        // 4. 리빌트가 정상적으로 종료되거나 예외처리되면 점유한 락을 풀어줌
        redisTemplate.delete(lockKey);
    }
  }
```

---

> [!WARNING] 랭킹 설계 핵심
> **동시성 제어 및 다중 서버 멱등성**
> 현업에서는 로컬 서버 1대로 테스트할 땐 스케줄러가 잘 동작하다가, 프로덕션에 서버를 3대 띄웠을 때 똑같은 알림이 3개 오거나 쿼리가 3번 나가는 참사를 자주 겪습니다. `setIfAbsent` (SETNX 형태) 명령어 1줄로 우아하게 선착순 1대의 서버만 작업을 위임하는 분산 락 패턴은 매우 중요합니다.
