# 섹션 4: 관리자 리스트 검색 + 필터 + 페이지네이션 공통 패턴

**[정합성 검증 완료 사항]**
> 사용자가 제시한 `AdminSearchRequest` DTO나 별도의 글로벌 `Specification` 클래스는 시스템에 존재하지 않습니다!
> 그 대신 DayPoo 백엔드는 **매우 세련된 인라인 JPA Criteria Builder (동적 쿼리) 방식과 Spring Data 명세인 `@PageableDefault`**를 혼용하는 깔끔한 클린 아키텍처를 `AdminManagementService` 안에 응집시켜 놓았습니다.

## 1. 컨트롤러: 파라미터 매핑 및 Pageable 주입

따로 공통 DTO를 만들지 않고 유연함을 위해 `GET` 요청의 Query Parameter로 필터를 개별 수신함과 동시에 페이징을 처리합니다.

```java
// AdminUserController.java
@GetMapping
public ResponseEntity<Page<AdminUserListResponse>> getUsers(
    @RequestParam(required = false) String search,               // 이메일 또는 닉네임 검색
    @RequestParam(required = false) Role role,                   // ROLE_USER 필터 
    @RequestParam(required = false) SubscriptionPlan plan,       // 구독제 플랜 필터
    @PageableDefault(size = 20, sort = "id", direction = Sort.Direction.DESC) Pageable pageable
) {
    return ResponseEntity.ok(adminManagementService.getUsers(search, role, plan, pageable));
}
```

## 2. 서비스: 동적 쿼리 (인라인 Specification 작성 패턴)

`MyBatis`의 동적 `if` 쿼리문처럼, 스프링 데이터 JPA가 지원하는 `Specification` 인터페이스를 즉席에서 람다식으로 구축하여 조건이 있을 때만 횡단 쿼리(`Where`)에 집어넣는 현대식 패턴을 사용합니다.

```java
// AdminManagementService.java 내부

@Transactional(readOnly = true)
public Page<AdminUserListResponse> getUsers(
    String search, Role role, SubscriptionPlan plan, Pageable pageable) {
    
    // 동적 쿼리 빌더
    Specification<User> spec = (root, query, cb) -> {
        List<Predicate> predicates = new ArrayList<>();

        // 1. 키워드 검색 (OR 조건) - %LIKE% 기반
        if (search != null && !search.isBlank()) {
            predicates.add(
                cb.or(
                    cb.like(root.get("email"), "%" + search + "%"),
                    cb.like(root.get("nickname"), "%" + search + "%")
                )
            );
        }

        // 2. 역할 (Role) 완전 일치 검색
        if (role != null) {
            predicates.add(cb.equal(root.get("role"), role));
        }

        // 3. 서브 쿼리(SubQuery)를 활용한 구독 여부 동적 필터링
        if (plan != null) {
            if (plan == SubscriptionPlan.BASIC) {
                // EXISTS NOT 로직 체인. 유료 구독에 해당 안되는 사람만 추출
                /* ...생략... */
            } else {
                // JOIN을 활용해 해당 플랜을 갖춘 사람 추출
                Join<User, Subscription> subJoin = root.join("subscriptions");
                predicates.add(cb.equal(subJoin.get("plan"), plan));
                predicates.add(cb.equal(subJoin.get("status"), SubscriptionStatus.ACTIVE));
            }
        }
        
        // 최종 where And() 감싸서 반환
        return cb.and(predicates.toArray(new Predicate[0]));
    };

    // Page<User>로 반환된 객체를 map() 함수를 통해 ResponseDTO로 일괄 변환 (은닉성 유지)
    return userRepository.findAll(spec, pageable).map(user -> AdminUserListResponse.builder()...build());
}
```

---

> [!TIP] 관리자 설계 팁
> **QueryDSL 도입의 대안 비교** <br>
> 위와 같은 `Criteria Builder` 패턴은 자바 표준이긴 하나 코드가 다소 장황해 지는(가독성 하락) 단점이 있습니다. 이 때문에 대규모 백오피스 스타트업 90% 가량은 `QueryDSL`이라는 코드 제너레이션 기반 QClass 라이브러리를 통해 필터 로직을 작성합니다. 추후 검색 기능이 10종류 이상으로 뻗어나간다면 마이그레이션을 고민해보는 것도 좋습니다.
