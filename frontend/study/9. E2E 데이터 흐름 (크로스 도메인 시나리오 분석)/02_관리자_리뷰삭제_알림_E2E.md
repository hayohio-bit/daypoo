# 시나리오 2: 관리자 신고 리뷰 삭제 및 알림 전파 E2E 분석

**[통합 핵심 포인트]**
> 백오피스 컴포넌트에서의 Optimistic UI 삭제 처리와 백엔드의 하드 삭제 후 서버간 Pub/Sub 알림이 프론트로 파급되는 흐름입니다.

## 1. 관리자의 UI 제어 (`AdminPage.tsx`)
1. 관리자는 대시보드의 `AdminPage.tsx` 내에 속한 화장실 리뷰 관제 탭에서 부적절한 사용자 리뷰를 적발합니다.
2. 휴지통 아이콘을 누르면, **Framer Motion**으로 설계된 삭제 다이얼로그(`DeleteConfirmDialog` 역할의 UI 블록)가 팝업됩니다.

## 2. 삭제 시도 및 API 호출
3. 삭제 확인 시 `AdminToiletController` (리뷰 파트는 `ToiletReviewController` 혹은 `AdminController` 파생 모듈)의 DELETE `api/v1/toilets/reviews/{id}` 엔드포인트로 `axios` 통신이 이루어집니다.
4. 백엔드는 JWT 토큰에서 `hasRole("ADMIN")`을 인가하고 해당 리뷰 엔티티를 찾은 후 제거합니다.

## 3. 알림 서비스 연동 (`ToiletReviewService.java`)
5. 리뷰 강제 삭제 완료 직후, 해당 `Review` 엔티티의 작성자(`user_id`)를 추적합니다.
6. 백엔드는 `NotificationService.send()`를 호출하여 `NotificationType.SYSTEM`으로 불건전 게시물 소거 통보를 발송합니다.
7. **[주의]** 현재 트리엔 `AdminService`에서 직접 알림을 보내는 것이 아닌 `ToiletReviewService` 또는 `NotificationService`에서 독립 수행되는 방향을 띱니다. 

## 4. 프론트엔드 알림 수신
8. 서버에서 발송된 SSE Event를 프론트엔드의 `<NotificationSubscriber />` 컴포넌트가 Listening 합니다. *(폴링 미사용!)*
9. 즉시 Header 중앙 벨 아이콘 위치의 `unreadCount`가 증가하며, 우측 상단이나 하단 Toast 컴포넌트를 통해 관리자의 시스템 경고임을 알립니다.

### E2E 매핑 테이블

| 레이어 | 담당 파일 / 컴포넌트 | 역할 요약 |
| :--- | :--- | :--- |
| 프론트 UI (관리자) | `AdminPage.tsx`, (다이얼로그 파편) | 리뷰 리스트 감지 및 Optimistic 화면 삭제 |
| 백엔드 코어 (관리) | `AdminToiletController` 등 리뷰 파트 | DELETE 매핑 및 권한 인가 |
| 백엔드 서비스 | `ToiletReviewService.java` | 물리적 DB Delete + 로깅 |
| 백엔드 알림 | `NotificationService.java` | 작성자에게 SYSTEM 사유 알림 저장 및 푸시 |
| 사용자 프론트 | `NotificationContext.tsx` | SSE 갱신 기반 상태(벨/뱃지) 동기화 |

### 에러 시나리오
*   **동시성 삭제 문제:** 관리자 A와 B가 동시에 같은 리뷰를 삭제 시도할 때, 두 번째 통신은 `404 Not Found` 혹은 `EntityNotFoundException`을 뱉습니다. 이때 백오피스는 전역 설정된 React의 Error Boundary나 `axios interceptor`에서 "이미 삭제된 항목" 토스트 텍스트를 띄웁니다.
*   **작성자 탈퇴 건:** 리뷰 작성자가 이미 탈퇴하여 `User`가 `null`인 경우, 알림 발송 로직(`NotificationService.send`) 연산 전 Null 체크(NPE 방어)를 수행해야 백엔드 크래시를 피할 수 있습니다.
