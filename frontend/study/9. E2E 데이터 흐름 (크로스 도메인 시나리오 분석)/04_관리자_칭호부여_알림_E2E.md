# 시나리오 4: 관리자 수동 칭호 제어 E2E 설계 분석

**[통합 핵심 포인트]**
> 이 시나리오는 기획안에 존재했으나 **실제 DayPoo 코드베이스엔 미구현(설계 예시)** 상태입니다. 현재 백엔드는 사용자가 행동 요건(방문 횟수 등)을 달성할 때 이벤트 리스너를 거쳐 `TitleAchievementService`에서 **자동으로 칭호를 부여**하고 알림을 쏘는 우수한 "Passive 로직"으로 동작 중입니다.

따라서 기획하신 기능(수동 개입)의 **신규 도입에 대비한 가이드 시나리오** 형태로 분석을 제시합니다.

## 1. 관리자의 타겟 식별 (기능 제안)
1. 백오피스 대시보드의 `AdminPage` / **Users Tab**에서 관리자가 포인트를 많이 모았거나 상위 랭커인 유저를 필터링(Specification 검색 파라미터 적용)합니다.

## 2. 부여 API 요청 (미구현 통신)
2. `[미구현]` 프론트엔드 목록 우측에 '수여하기' 버튼을 추가하여 클릭합니다.
3. 백엔드 `AdminTitleController.java`에 미비된 단일 엔드포인트 `POST /api/v1/admin/titles/{id}/grant?userId={userId}` 를 설계 및 호출합니다.

## 3. 백엔드 트랜잭션 및 알림 트리거 수행 
4. 서비스 컴포넌트(e.g., `AdminManagementService.java` 내부)에서 `UserTitleRepository`에 권한 레코드를 `INSERT` 합니다.
5. 직후 `NotificationService.send()` 를 통해 `NotificationType.ACHIEVEMENT` 등급으로 메시지를 주입합니다.
   - 예: `"특별 관리자 칭호 [황금빛 배변자]를 수여받았습니다!"`

## 4. 클라이언트 즉시 반영
6. `NotificationSubscriber`가 SSE 채널로부터 위의 텍스트 페이로드를 즉각 다운스트림 받습니다.
7. 축하 토스트 팝업이 노출되며 사용자 UI(프로필/헤더 영역)에 반짝이는 새 뱃지가 `Framer Motion` 전환 애니메이션과 함께 등장합니다.

### 도입을 위한 매핑 트리 구조 (추천안)

| 레이어 대상 | 필요 작업 내역 (How to Implement) |
| :--- | :--- |
| **`AdminPage.tsx`** | Users 테이블 UI 컬럼의 'Action' 영역에 뱃지 다중 선택 Modal 창을 추가 구성. |
| **`apiClient.ts`** | `grantTitleAdmin(titleId, userId)` 프로미스 호출 모듈 작성. (RTK Query 도입 시 Mutation 파트 추가 필요) |
| **`AdminTitleController`** | 백엔드 내에 누락된 `@PostMapping("/{id}/grant")`를 생성하고 권한 및 `userId` 매핑. |
| **`AdminManagementService`** | 부여(`UserTitle` 연관 관계 매핑) 및 중복 지급 필터(`existsByUserIdAndTitleId`) 로직 포함. 자동 저장. |
| **`NotificationContext`** | 기존 구축된 SSE 연결 파이프라인에서 타입 인식 확장. |

### 설계 병목 및 예방 시나리오
*   **중복 부여 허용 취약점**: 관리자가 UI 창에서 연속 클릭 버그(Debounce 누락)를 발생시켰을 시, 다중으로 `INSERT`가 일어날 수 있습니다. 백엔드 레벨에서 `UserTitle`에 대한 `(user_id, title_id)` 복합 유니크 키(Composite Unique Key)를 반드시 설정해야 DB 정합성이 엇나가지 않습니다.
*   **대규모 일괄 지급(Batch Grant) 다운타임**: 한 명의 유저가 아니라 등급 필터로 1,000명의 상위 랭커에게 단체 부여할 경우, 1,000번의 `Notification` Insert 쿼리가 발생해 성능이 죽습니다. `@Async` 또는 Bulk Insert 처리를 잊지 마세요.
