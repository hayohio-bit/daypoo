# 시나리오 5: 테라폼 기반 봇 시뮬레이션 및 데이터 시딩 E2E 분석

**[통합 핵심 포인트]**
> 현재 프로젝트에는 `.github`, `terraform`, `scripts` 디렉토리가 일부 존재하나, 기획하신 `[simulation/bot/scenario/]` 형태의 `Terraform Lambda` 기반 **봇 파이프라인은 리포지토리 구성 상 미구현 상태**입니다. (대신, 관리자 패널의 `/api/v1/admin/generate-test-data` 컨트롤러를 바탕으로 수동 더미 주입이 가능합니다.)
>
> 앞으로 **대규모 트래픽 부하 테스트**를 위해 확장할 클라우드 시점의 E2E 인프라 파이프라인을 제안합니다.

## 1. 인프라 클라우드 트리거 (Lambda & Terraform)
1. **[미구현 제안]** AWS EventBridge Cron 스케줄에 의해 테라폼으로 프로비저닝된 `bot_lambda`가 특정 시간대 통계를 증폭시키기 위해 트리거를 시작합니다.
2. 봇 워커들이 백엔드 Nginx/ALB 배포 URL을 치면서 다수의 유저 세션으로 로그인(`POST /api/v1/auth`)을 맺고 동시 100건 이상의 `POST /api/v1/records` 트래픽 난사를 발생시킵니다.

## 2. 데이터 세딩 가속 (RankingDataSeeder 대체)
3. 대규모 데이터가 들어올 때, 백엔드의 `PooRecordEventListener` 시스템이 비동기(@Async)로 랭킹 `ZSet` 점수를 조우하여 올립니다.
4. 그러나 트래픽이 뚫리지 않도록 랭킹 일일히 연산하는 것이 낭비이므로, 실제 로드 테스트 시나리오에서는 **RankingDataSeeder**(관리자 전용 대규모 Redis PipeLine 벌크 갱신 모듈)를 직접 가동해 순식간에 데이터를 강제 조작합니다. (`AdminController.rebuildRankings()` 와 유사 컨셉)

## 3. 관리자 대시보드 리얼타임 모니터링 (`AdminPage.tsx`)
5. 관리자가 접속한 대시보드는 1분 혹은 수 초 주기로 `apiClient.get('/api/v1/admin/stats')` 동기화를 거쳐 서버 데이터를 끌어옵니다. (RTK Query Polling 60000ms 방식 부재로 현재는 수동 컴포넌트 갱신)
6. 봇의 데이터 폭증이 집계 인덱스나 캐시에서 취합되어 반환되면, 그라데이션 SVG와 타임라인 구조를 가진 **`Recharts ComposedChart`** 곡선 막대가 폭발적으로 솟구치며 관리자에게 "비정상 / 정상 대규모 트래픽 유입" 사실을 시각화합니다.

### E2E 매핑 방안 (아키텍처 확장 플랜)

| 클라우드 / 코드 스택 | 설계 목적 / 구현 위치 예시 |
| :--- | :--- |
| **`Terraform (.tf)`** | `lambda.tf`, `eventbridge.tf` 구성. Node.js 혹은 Python 기반 Axios 봇 빌드 컨테이너화 |
| **비동기 Seeder 컨트롤러** | 관리자 권한 API로 이식 -> `AdminDataSeederService` 형태로 더미 레코드를 대량 `saveAll()` |
| **`PooRecordRepository`** | 벌크 인서트 시 Sequence Lock-out 이슈 대비를 위한 JDBC 템플릿(JdbcTemplate) bulk-insert 변경 구상 |
| **`DashboardView` (React)** | 폴링 주기를 설정하여 트래픽 추적기용 실시간 `Recharts` 라이브 라인 플롯 변형 추가 |

### 에러 시나리오 파악
*   **DDoS 오인에 의한 Rate Limit 밴 현상**: 글로벌 AOP인 `RateLimitAspect.java`는 봇이 발생시키는 초당 수십 건의 트래픽을 비정상으로 간주해 HTTP 429 에러를 뱉어버립니다. 시뮬레이션 환경이 실제 운영 레벨의 AOP에서 차단되는 딜레마를 겪으므로, 봇 테스트 전용 IP 대역이거나 테스트 토큰에 대한 `RateLimit` Bypass 검사 필터를 개발해 놓아야 합니다.
*   **Redis ZSet Throttling (스레드 병목)**: 만 대의 기기에서 일시다발적으로 `ZINCRBY`가 걸리면 Redis Event Loop가 버벅이므로, Bulk Add `ZADD` 기반의 캐시 파이프라이닝 최신화 배수관로(PipeLining) 사용을 고려하십시오.
