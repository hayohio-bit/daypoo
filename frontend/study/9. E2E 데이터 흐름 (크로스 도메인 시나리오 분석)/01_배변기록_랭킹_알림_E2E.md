# 시나리오 1: 배변 기록 생성부터 실시간 랭킹 추적 E2E 분석

**[통합 핵심 포인트]**
> 사용자가 화장실에서 기록을 생성하는 즉시 비동기로 랭킹 ZSet이 업데이트되며, 폴링 오버헤드 대신 **Server-Sent Events(SSE)**를 통해 프론트엔드로 즉각 알림이 전파되는 Full-Reactive 사이클입니다.

## 1. 프론트엔드 - 기록 작성 (`RecordPage.tsx`)
1. 사용자가 Bristol 척도, 색상, 특이사항 등을 기입하고 이미지를 첨부합니다.
2. 폼 내에서 `axios` 래퍼인 `api.post('/api/v1/records')`를 호출합니다.
   *(주의: 기획상의 RTK Query는 미사용 상태이며 Context API + Local State로 관리됨)*

## 2. 백엔드 - 서버 수신 및 AI 분석 (`PooRecordService.java`)
3. `PooRecordController`가 요청을 받아 `PooRecordService`로 전달합니다.
4. 기록이 트랜잭션 내에서 `PooRecord` 엔티티로 영속화됩니다.
5. 이미지 첨부 시 `AiClient.analyzePoopImage()`를 통해 Python 서비스(`ai-service` 서브모듈)의 멀티파트 AI 분석(`POST /api/v1/analysis/analyze`)을 동기 호출해 점수를 부여받습니다.

## 3. 이벤트 기반 아키텍처 - 랭킹 반영 (`PooRecordEventListener.java`)
6. 메인 트랜잭션이 성공하면 커밋 직후(`AFTER_COMMIT`), 스프링 애플리케이션 콘텍스트에 `PooRecordCreatedEvent`가 발행됩니다.
7. `@Async`로 대기 중이던 **`PooRecordEventListener`**가 이벤트를 낚아챕니다.
   - `user.addExpAndPoints()` 실행
   - `rankingService.updateGlobalRank()` 실행 -> Redis의 `ZINCRBY global_ranking` 원자적 증분

## 4. [설계 예시] 순위 변동 알림 처리
8. **[미구현 / 설계 제안]** 현재 `RankingEventListener`에서 랭킹을 추적해 `Notification`을 넣는 구조는 완전히 구현되지 않았습니다. 현재는 점수 연산 직후 "칭호(Title) 획득 여부"만 판별(`TitleAchievementService.checkAndGrantTitles`)하여 칭호 획득 시 알림을 보냅니다.
   - **이상적인 구현 방안:** `ZREVRANK` 명령어로 이전 등수와 캐싱된 등수를 비교, 만약 상위 100위 진입 시 `NotificationService.send()` 호출

## 5. 프론트엔드 - 실시간 반영 (`NotificationSubscriber.tsx`)
9. 서버에서 생성된 `Notification` 엔티티는 곧바로 Redis Pub/Sub을 거쳐 서버의 `SseEmitter`를 통해 프런트엔드로 1초 이내 전송됩니다.
10. 프론트엔드의 `NotificationSubscriber`가 SSE 메시지를 파싱하여 `unreadCount`를 갱신하고, `Framer Motion`이 적용된 토스트 알림 팝업을 띄워 사용자에게 랭킹 변동을 실감케 합니다.

### E2E 매핑 테이블

| 레이어 | 담당 파일 / 컴포넌트 | 역할 요약 |
| :--- | :--- | :--- |
| 프론트엔드 UI | `RecordPage.tsx`, `RecordForm` | 폼 구성 및 FormData 생성 |
| 프론트엔드 통신 | `apiClient.ts` | Bearer Token 주입, Error 가로채기 |
| 백엔드 API | `PooRecordController.java` | HTTP -> 서비스 파라미터 맵핑 |
| 백엔드 코어 | `PooRecordService.java` | DB Insert + AI 동기화 호출 + 이벤트 발행 |
| 백엔드 AOP | `PooRecordEventListener.java` | 비동기 Redis 랭킹 점수 갱신 + 칭호 수여 |
| 알림 계층 | `NotificationService.java` | 칭호 혹은 랭킹 알림 SSE 발사 |
| 프론트엔드 소켓 | `NotificationSubscriber.tsx` | 알림 수신 후 Context 갱신 및 Toast 트리거 |

### 에러 시나리오 및 성능 예상
*   **AI 이미지 분석 타임아웃 (`AiClient` 응답 실패):** `AiClient.java` 내의 `@Retryable` 아키텍처가 1초 Backoff 간격으로 2회 재시도합니다. (총 4초 대기). 실패 시 디폴트 점수(Basic 10점 등) 부여 흐름으로 빠져야 사용성을 해치지 않습니다.
*   **응답 성능 예상치:** AI 미사용시 80ms 내외, AI 시각 모델 통신 개입 시 2~4초 소요. 이벤트 발행 아키텍처 덕분에 랭킹/알림 로직(20ms)은 메인 응답 스레드에 병목을 주지 않습니다.
