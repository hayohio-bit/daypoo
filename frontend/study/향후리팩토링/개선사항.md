## 2. 유효한 개선 제안 (도입 적합성 판단)

| # | 제안 항목 | 현재 상태 | 도입 적합성 | 이유 |
|---|----------|----------|------------|------|
| 1 | **RTK Query 도입** (`폴더 2, 4, 8`) | `@reduxjs/toolkit` 미설치, useState + fetch 패턴 | ✅ 적합 | 탭 전환마다 API 재호출 발생 — 캐싱·폴링 자동화로 네트워크 요청 50% 이상 감소 가능 |
| 2 | **서비스 모듈 분리** (`폴더 2/02`) | `reviewService.ts` 1개만 존재, 나머지 훅에서 직접 `api.get()` 호출 | ✅ 적합 | 컴포넌트-API 결합도 제거 — 테스트 및 재사용성 향상 |
| 3 | **SSE_ENABLED 활성화** (`폴더 6/03`) | `false`로 하드코딩 비활성화 | ✅ 적합 (조건부) | 백엔드 SSE 엔드포인트(`/notifications/subscribe`) 이미 구현됨 — 연결 테스트 후 true 전환 |
| 4 | **Notification 테이블 인덱스** (`폴더 5/01`) | `@Index` 어노테이션 미적용 | ✅ 적합 | 알림 목록 조회(`WHERE user_id AND is_read=false`)가 풀스캔 — 복합 인덱스 추가로 쿼리 성능 개선 |
| 5 | **Notification 커서 페이지네이션** (`폴더 5/04`) | 전체 목록 List 반환 (페이지네이션 없음) | ✅ 적합 | 알림 누적 시 응답 페이로드 무한 증가 — `WHERE id < {cursor} LIMIT 20` 패턴 필요 |
| 6 | **DiceBear 아바타 가상화** (`폴더 4/05`) | 전체 랭킹 아바타를 한 번에 렌더링 | ✅ 적합 | `createAvatar().toDataUri()` 동기 호출이 100+ 항목 시 메인 스레드 블로킹 — `react-window` 도입 필요 |
| 7 | **AdminPage.tsx 분리** (`폴더 8/01`) | 4186줄 단일 파일, 탭 전환은 조건부 렌더링 | ✅ 적합 (중장기) | 파일 크기로 인한 IDE 응답 저하 및 코드 리뷰 어려움 — React Router 중첩 라우트로 점진적 분리 권장 |
| 8 | **DataTable 공통 컴포넌트** (`폴더 8/03`) | 각 탭(Users, Toilets, Shop 등)에 인라인 `<table>` 반복 | ✅ 적합 | 검색·정렬·페이지네이션 로직 중복 5회 이상 — `DataTable<T>` 제네릭 컴포넌트로 추출 |
| 9 | **Zod 폼 유효성 검증 도입** (`폴더 8/04`) | `zod` 미설치, 직접 `if` 분기로 검증 | ✅ 적합 | 관리자 아이템 생성·수정 폼에 필드 10개 이상 — Zod 스키마로 에러 메시지 일원화 |
| 10 | **rankingService.ts 추출** (`폴더 2/02`) | `useRankings.ts`에서 직접 `api.get('/rankings/global')` 호출 | ✅ 적합 | 서비스 레이어 없이 훅이 API 직접 호출 — `rankingService.ts` 분리 시 RTK Query 전환도 용이 |
