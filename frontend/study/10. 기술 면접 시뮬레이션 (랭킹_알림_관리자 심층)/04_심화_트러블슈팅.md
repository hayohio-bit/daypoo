## Q25. [심화] 프로젝트 진행 중 겪은 가장 까다로웠던 기술적 과제와 돌파 과정
- **핵심 답변 포인트**
  - Redis 랭킹 동점자의 선착순 무력화 문제 파악
  - 비즈니스 도메인의 한계를 수학적 구조로 도해(해결)한 능력
  - 프론트 상태(RTK vs Spring 이벤트)의 부조화 극복 과정
- **모범 답변**
  가장 큰 두통은 인메모리 Redis의 특성으로 야기된 '랭킹 선착순 역전 현상'과, 프론트/백엔드 알림 속도의 불균형(Race Condition)이었습니다. Redis `Sorted Set` 체제는 점수가 똑같을 때 키의 알파벳 사전순으로 편가르기를 하기 때문에 먼저 땀 흘려 점수를 쌓은 사용자가 우위를 잃는다는 치명타가 있었습니다. 이를 해결하고자 점수에 도달 시간을 역산한 Timestamp 소수점을 붙이는 부동소수점 인코딩 트릭을 구현해 냈습니다. 이로인해 별도 RDBMS 정렬 자원을 낭비하지 않고도 단일 트랜잭션 내에서 완벽한 차순 방어가 가능해져, 팀원 모두가 만족하는 데이터 불변성의 아키텍처를 확보해 냈습니다.
- **꼬리 질문 예시**
  1. 만일 그 동점 타임스탬프 조차 한 치의 오차 없이 동시에 요청되었다면 분산 환경에선 어떻게 락을 겁니까?
  2. 트러블슈팅 시 문제를 분해하는 접근 방식 자체에 대해 예를 들어 주실 수 있나요?
- **평가 기준**
  특정 기술 도구의 사용법을 넘어 그것의 고착화된 버그나 로직의 한계 상황을 엔지니어링 적으로 돌파해 낸 수리적/객체지향적 집요함을 봅니다.

## Q26. [심화] 랭킹 시스템의 확장성(Scalability) 100배 증가를 고려한 후속 재설계 복안
- **핵심 답변 포인트**
  - 단일 Redis 인스턴스의 병목 해소 전략 (Redis Cluster / Redis Sentinel)
  - 파티셔닝(샤딩) / CQRS 패턴을 필두로 한 읽기-쓰기 분리
  - 이벤트 소싱 및 메시지 큐(RabbitMQ, Kafka) 파이프라인
- **모범 답변**
  앱이 스케일아웃 하여 초당 10만 랭킹 갱신 트래픽이 발생한다면 현재의 단일 Redis 구조는 SPOF(Single Point of Failure) 취약점이 도드라집니다. 트래픽 확장을 고려한다면 첫 번째로 Redis Cluster를 탑재하여 키 스페이스 파티셔닝(샤딩)을 해야 합니다. 둘째로 극심하게 분기 되는 데이터 삽입(명령)부와 데이터 조회(결과)부를 찢는 CQRS 패턴을 고도화하여, RDBMS 쓰기용 노드와 Redis 랭킹 리딩 노드의 궤도를 완벽히 나누고 싶습니다. 아울러 카프카(Kafka) 등 MQ를 이용한 대규모 분산 보장 이벤트소싱 기법을 중간에 두어, 랭킹 쓰레드가 일시 중단되더라도 데이터 유실 없이 순차 파이프라이닝 복구가 일어나게 할 것입니다.
- **꼬리 질문 예시**
  1. CQRS 분리 환경 내 노드 간 데이터 지연 동기화 시간(Eventual Consistency)으로 인한 UI 불이치는 어떤 화면 처리가 병행되어야 합니까?
  2. Redis Cluster 구축에서 Hash Slot 원리를 알고 계신가요?
- **평가 기준**
  지원자가 현재의 우물(단일 모놀리식 서버)에 갇히지 않고 거대한 대유량 분산 MSA 섀도를 그릴 수 있는지 검증합니다.

## Q27. [심화] 알림 시스템을 완전한 SSE 체계로 전환할 실시간 설계 방법론은?
- **핵심 답변 포인트**
  - `SseEmitter` 및 접속 자원(`ConcurrentHashMap`) 관리 체계 통달력
  - 유의 깊은 Connection 포트 관리 & 타임아웃 해지(방치 회피)
  - 연결 유실 시의 Retry 재연결 전략 구비
- **모범 답변**
  향후 RTK Polling 방식이 한계에 봉착해 백엔드 자원을 SSE(Server-Sent Events)로 상향 평준화할 경우, 안전한 커넥터 풀이 생명입니다. Spring 백엔드에서 접속자의 `SseEmitter` 객체들을 스레드 세이프(Thread-Safe)한 `ConcurrentHashMap`에 매핑할 것입니다. 그러나 알림 구조상 방치되는 연결이 많아 OOM(Out of Memory)과 포트 좀비 현상이 생기기 쉬우므로 타임아웃 제한과 명시적인 `onCompletion()`, `onTimeout()` 콜백 해제 매커니즘을 꼼꼼하게 삽입할 것입니다. 더불어 프론트엔드의 `EventSource` 단독 사용 시 네트워크 드랍 오류가 잦으므로 `Reconnect(fallback)` 무한 루프 폭탄을 방지할 지수 백오프 기반 재접속 전략을 욱여넣어야 합니다.
- **꼬리 질문 예시**
  1. SseEmitter 타임아웃 만료 시 브라우저 측에서 503 에러 코드가 뜰 수 있는데 503 HTTP 전구간 해결책이 있나요?
  2. 스레드 세이프하다고 해서 ConcurrentHashMap 상에 수백만 컬렉터가 쌓이면 위험한데, 어떻게 정리할 건가요?
- **평가 기준**
  웹 기반 지속 네트워크의 위험성과 자원 관리 해제라는 중요한 CS 펀더멘털을 파악합니다.

## Q28. [중급] TypeScript 프론트에서 Back-End 자바 DTO 스키마 불일치를 해결하는 법
- **핵심 답변 포인트**
  - 수동 매핑(타입 선언)의 휴먼 에러 결함 지적
  - `openapi.yaml` (Swagger) 기반의 타입 생성기 엑세스
  - `Zod` 등의 런타임 스키마 방어벽 검증력 확보
- **모범 답변**
  프론트와 백이 극심하게 분업할 경우, 백엔드 Entity나 DTO 필드명(ex. `userName` -> `user_name`)이 하나라도 기습 변경되면 전체 앱이 렌더링을 뻗치는 심각한 타이핑 동기화 오류를 겪었습니다. 저는 이를 해결하고자 프론트에서 타입을 일일이 쳐 넣지 않고 백엔드의 Swagger(OpenAPI) 문서의 `yaml` 명세서를 추출한 후 `OpenAPI Generator` 플러그인 도구로 TypeScript Interface들을 찍어내는 자동화 파이프라인 동기화를 진행했습니다. 또한 외부망이나 꼬인 타입으로 들어올 확률이 있는 치명적인 관리자 도메인은 프론트 구동 런타임에서 막아낼당할 `Zod` 스키마 체커를 두어 방탄(Type-Safe) 수준을 영구 확장했습니다.
- **꼬리 질문 예시**
  1. Zod 라이브러리로 런타임 검증을 자주 처리할 시 프론트 성능에 미치는 패널티/부하는 없는가요?
  2. 백엔드 배포 직후 CI 파이프라인 상에서 프론트 타입을 자동으로 당겨가는 액션을 연동하진 않았나요?
- **평가 기준**
  생산성이 지배하는 타입-세이프 파이프라인과 프론트엔드/백엔드 불일치 커뮤니케이션 극복성을 확인합니다.

## Q29. [심화] 관리자 도메인 오픈에 따르는 보안 위협과 방지 마지노선 구축
- **핵심 답변 포인트**
  - URL 기반의 강경한 RBAC (Role-Based Access Control) 인가 방어막
  - XSS / CSRF 공격 주입 방어 (Stateless 포함)
  - 관리자 감사 로그(Audit)와 IP 화이트리스트 도입
- **모범 답변**
  Admin 백오피스는 유저 권한은 물론 포인트나 결제 조작도 가능한 최상위 폭탄입니다. 무조건 1차 진입선인 Spring Security 필터단(`SecurityFilterChain`) 에서 `/api/v1/admin/**` 의 루트는 Enum으로 직조된 `hasRole('ADMIN')` RBAC 구조를 걸어 허가되지 않은 일반 침투를 근본 차단했습니다. 두 번째 위협인 XSS 스크립트 오염(리뷰 게시판 경유) 및 세션 탈취 공격(CSRF)은 React 특유의 이스케이프 방어와 서버 `Stateless JWT` 토큰 설계로 회피했습니다. 마지막으로 사내망 오염 방지를 위해 추후 운영 방벽인 IP 화이트리스트 접근 제한을 걸어두고 철저한 AOP 감사 이력(Audit Log) 기반의 누가 무엇을 삭제/변경했는지 기록하는 추론 아키텍처를 구현 완료했습니다.
- **꼬리 질문 예시**
  1. 탈취된 JWT 토큰으로 해커가 접속을 이어갈 때 이를 방지하는 백엔드 Invalidation 전략(Redis 블랙리스트)는 작동하나요?
  2. 왜 CSRF 공격은 JWT가 적용된 Stateless 상태에서는 비교적 방어율이 높습니까?
- **평가 기준**
  웹 애플리케이션의 OWASP 기본 보안 지식 및 백오피스 관리에 대한 철저한 아키텍트 마인드를 살핍니다.

## Q30. [심화] DayPoo 이 프로젝트를 수만 명의 C1c 환경, AWS 프로덕션에 구축하기 위한 최종 인프라 시안
- **핵심 답변 포인트**
  - AWS ECS 또는 EKS 기반의 오케스트레이션 부하 대응 컨테이너 아키텍처
  - Multi-AZ RDS 및 백업 클러스터 인/아웃 방식
  - CloudFront(CDN), ALB 트래픽 라우팅 분산 구조망
- **모범 답변**
  현행 Docker 프레임워크 이상의 확장으로 무중단 운영과 트래픽 대응을 고도화 하려면 AWS 기반 프로덕션 인프라 청사진을 그려야 합니다. 사용자의 트래픽은 `Route 53`과 `CloudFront`(CDN)를 통과하며 캐싱 성능과 보안(WAF) 혜택을 챙겨 얻은 뒤, 앞단의 프론트 및 `Application Load Balancer(ALB)` 스캐너로 뿌려집니다. 백엔드 Spring 컨테이너들은 `ECS` 파게이트나 `EKS` 상에서 Auto-Scaling 그룹화되어 트래픽 변동에 자율 증식합니다. 데이터베이스는 단일 장애 포인트를 줄이기 위해 `RDS Multi-AZ` 구조의 이중화를 취하고, 캐싱용 `ElastiCache (Redis)`를 결박하여 병목 없는 강력한 수만 동시접속(10K) 레이턴시 무중단 시스템 환경망을 조율해 낼 것입니다.
- **꼬리 질문 예시**
  1. Scale Out이 폭주할 때 유저들의 세션 또는 데이터 멱등성은 어떻게 유지 보장하나요?
  2. CloudWatch 등을 활용한 모니터링 경보(Alert) 시스템 구축 관록은 있으신가요?
- **평가 기준**
  이론과 개발을 뛰어넘어, 대규모 엔터프라이즈 운영 시 발생할 분산 처리 및 고가용성 네트워크 생태를 머릿속에 도식할 수 있는지를 판별합니다.
