## Q17. [기초] 3-tier 환경/AI 분리 아키텍처를 구성한 이유는 무엇인가요?
- **핵심 답변 포인트**
  - 확장성과 도메인 관심사 분리
  - Python AI 코어 모듈의 언어적 강점(LangChain, FastAPI)
  - 스프링 부트 서버의 자원 소모 방벽 역할
- **모범 답변**
  시스템 구조 설계 시 백엔드 Spring 서버 안에 모든 로직을 욱여넣지 않게 설계했습니다. React 프론트-Spring 백엔드 간 통신 외에도, 무거운 연산이 일어나는 LLM AI 분석 서버를 Python + FastAPI 컨테이너로 독립시켰습니다. 생성형 AI 특성상 Python 환경인 LangChain 파이프라인 구성이 절대적으로 필요했고, 스프링 애플리케이션의 메모리와 스레드가 AI 리포트 변환 대기로 인해 멈추는(Bottle Neck) 부하 전이 현상을 끊어내기 위함이었습니다. 향후 생성형 분석 트래픽이 많아질 경우 AI 노드만 별도로 Scale-Out 할 수 있는 아키텍처적 유연함을 확보했습니다.
- **꼬리 질문 예시**
  1. 외부 AI 서버로 이미지를 넘길 때, 직접 Base64로 넘기는 것과 S3 URL을 넘기는 것의 차이는 무엇입니까?
  2. 서버 통신 간 Circuit Breaker (서킷 브레이커)에 대해 설명해 주세요.
- **평가 기준**
  MSA로 나아가는 중간 단계 아키텍처 확장성 및 물리적인 언어/서버 환경 분리의 논리적 근거를 파악합니다.

## Q18. [중급] 백엔드와 프론트엔드 통신 간 마주한 CORS 이슈 설계 전략
- **핵심 답변 포인트**
  - 도메인 간 Origin 보안 제약 극복
  - Spring의 `CorsConfig` 및 `allowCredentials` 속성 매핑
  - `Preflight` (OPTIONS) 요청 최적화 지식
- **모범 답변**
  로컬 서버나 프로덕션 운영 시, 프론트(예: GitHub Pages)와 백엔드 API 클러스터의 Origin 주소가 달라 브라우저 단에서 통신을 가로막는 CORS(Cross-Origin Resource Sharing) 위반 에러에 부딪혔습니다. 이를 해결하기 위해 백엔드에 `CorsConfigurationSource`를 구현하여, 특정 출처만 좁혀 허용(`allowedOrigins`) 하도록 화이트리스트 처리했습니다. 특히 우리는 쿠키 대신 Authorization 헤더를 쓴다고 해도, 간혹 JWT 다중 환경의 `credentials` 통신이나 각종 커스텀 토큰 교환을 위해 `allowCredentials(true)` 속성 옵션을 병행하여 인증 패킷이 드랍되지 않게 방어했습니다.
- **꼬리 질문 예시**
  1. `CorsRegistry`에서 모든 Origin(`*`)을 허용하면 `allowCredentials(true)`와 충돌하는 이유는?
  2. 브라우저가 본 요청을 보내기 전 OPTIONS로 사전 점검하는 Preflight Request 횟수를 최적화(`maxAge`)하는 방법은 무엇인가요?
- **평가 기준**
  웹 브라우저 보안 규약에 대한 이해와 네트워크 환경의 안전한 데이터 교류 구성 능력을 체크합니다.

## Q19. [심화] 배변 기록 생성부터 랭킹 반영, 알림 전달까지 전체 E2E 플로우
- **핵심 답변 포인트**
  - API Payload 및 JWT 가드 인증
  - 트랜잭션 DB CUD 무결성 
  - 백엔드 이벤트 기반 EDA 연결고리
  - 최종 UI 통보(SSE) 및 스토어 동기화 메커니즘
- **모범 답변**
  유저가 지도를 통해 인증 기록(Payload)을 전송하는 순간 전체 시스템의 E2E 파이프라인이 즉시 발동됩니다. 
  1) Spring Security 필터 체인을 거쳐 인증된 요청이 `PooRecordService` 로 진입해 RDBMS 커밋을 생성합니다. 
  2) 이후 랭킹 점수가 가산되며 Redis ZSet의 캐시 정렬이 변경(ZINCRBY) 됩니다. 
  3) 이 랭킹 변경 시의 변화량을 `RankingChangedEvent`로 캐치해 `@Async` 스레드 백그라운드 모델이 알림 큐를 트리거합니다. 
  4) 마지막으로 해당 알림 데이터는 `SseEmitter`를 통해 프론트엔드의 `NotificationContext` 연결망에 탑승하여 시스템 렌더링 피드를 밀어내며 유저에게 최종 변경값을 애니메이션으로 알립니다.
- **꼬리 질문 예시**
  1. 만일 트랜잭션 단계(2번 과정)에서 Redis 저장이 실패했다면 DB RDBMS 데이터는 어떻게 보상 롤백되나요?
  2. E2E 진행 중 AI 서버로부터의 분석 Response가 오지 않을 때 프론트는 어떤 컴포넌트 렌더링 처리를 하나요?
- **평가 기준**
  크로스 도메인에 도달하는 서비스 데이터 생명주기에 대한 Full-Stack 파이프라인의 입체적 이해도를 평가합니다.

## Q20. [심화] 실시간 알림을 Polling으로 구현하다가 SSE로 확장하는 전략
- **핵심 답변 포인트**
  - 단방향(SSE) vs 양방향(WebSocket) 패러다임 차이 입증
  - 가변적 HTTP 통신 주기 제어에 따른 네트워크 부하 이완
  - 복잡도 대비 투자 환산 및 도메인 특화
- **모범 답변**
  초기 애플리케이션의 실시간 알림 구조는 `RTK Query`를 이용한 간헐적 Polling 기법으로 개발 생산성을 높였습니다. 하지만 트래픽 증가에 따라 백엔드 무의미한 조회 부하가 증폭되어 서버 부하 방어를 위해 단방향 스트리밍 기술인 `SSE(Server-Sent Events)` 채택을 고민했습니다. 양방향인 WebSocket보다 인프라 비용(Connection 유지비용) 및 구현 난이도가 현저히 낮고, 알림 비즈니스 본질 자체가 '서버 단에서 변화를 밀어주는(Push)' 역할이 전부였기에 가장 압도적이고 군더더기 없는 도메인 주도 설계 기법이 될 거라 판단했습니다. 
- **꼬리 질문 예시**
  1. 잦은 Polling으로 일어나는 HTTP 헤더 오버헤드를 막는 방안에는 어떤 것들이 있나요?
  2. Nginx를 리버스 프록시로 사용 시 SSE 데이터 스트리밍이 버퍼링 결함으로 끊기는 이유와 우회 설정은 무엇입니까?
- **평가 기준**
  현재 기술 스택의 한계 비용(Trade-off)을 인지하며 적소의 라이브러리/프로토콜을 논리적 기준을 통해 판단하는지 봅니다.

## Q21. [중급] AI 서비스와 백엔드 간 통신 시 Redis 캐시를 24시간으로 설정한 이유
- **핵심 답변 포인트**
  - 장기 지연 시간(Lag) 방어 
  - GPT API 등 외부 서드파티 비용 절감 방벽
  - 주간/일별 통합 리포팅 성격에 기인한 데이터 라이프사이클 측정
- **모범 답변**
  LLM 기반의 분석 및 비전 이미지를 조합하는 추론 시간은 통상 최저 10초에서 최대 30초까지 육박하는 극단의 지연 속도를 낳습니다. 사용자가 응답을 얻고 나서 화면을 리프레시하거나 페이지를 왕복할 때마다 매번 이 API를 던지면 병목과 토큰 누적 결제 비용이 극심해집니다. 따라서 "오늘의 건강 상태" 라는 도메인 문맥 특징을 살려, 한 번 추론되어 반환받은 AI 결과값을 DB와 더불어 `Redis` 상에 보관하고 TTL(Time to Live)을 하루 24시간 치로 부여했습니다. 이후 들어오는 모든 동일 유저 데이터 요청은 AI 머신 개입 없이 in-memory 캐시에서 즉출되므로 0초대에 응답하는 마법을 부렸습니다.
- **꼬리 질문 예시**
  1. 24시간 사이에 배변 기록이 1건 더 추가되어 기존 AI 리포트 결과와 어긋날 정합성 문제는 어떻게 격리합니까?
  2. 사용자 1만 명이 동시다발적으로 만료된 캐시 구간에 몰릴 때 AI 서버를 향한 폭격을 막으려면(Cache Stampede 예방)?
- **평가 기준**
  외부 API 과금 및 지연 시간 방어와 관련한 거시적 인프라 캐싱 아이디어와 비즈니스 논리를 살펴봅니다.

## Q22. [심화] 봇 시뮬레이션 환경 구현과 Terraform 분리의 이점은?
- **핵심 답변 포인트**
  - 대량 로드(부하) 테스트 및 봇 트래픽 확보 
  - IaC(Terraform) 기반의 코드화 된 인프라 운영
  - Lambda 워커 환경을 통한 리소스 비용 해소
- **모범 답변**
  대규모 랭킹 기능(100만 사용자 동시 조회 스펙)이나 락(Lock) 충돌 결함을 모니터링하기 위해서는 광범위한 트래픽 베이스의 로드 생성이 필요했습니다. 저는 백엔드 내부의 더미 코드로 이 오버헤드를 생산하지 않고, 인프라 코드 프로비저닝인 `Terraform` 스크립트에 AWS Lambda 봇 시뮬레이터를 분리 구성하여 쏘는 인프라 벤치마킹 구조를 택했습니다. 운영 서버 자원을 축 내지 않고 오로지 테스트/시뮬레이션 목적으로만 가동되며 사용 시에만 클라우드 요금이 과금되는 Serverless 특성이 큰 시너지를 창출했습니다.
- **꼬리 질문 예시**
  1. 부하 생성 봇 트래픽이 Rate Limiting 보안 파이프라인에 걸려 튕길 경우 백엔드의 우회 방법(화이트리스트 등)은 어떻게 처리했나요?
  2. Terraform의 State 파일 관리 시 팀원 간 버전 충돌이나 Lock을 해결하기 위한 S3 백엔드 방식의 핵심은 무엇입니까?
- **평가 기준**
  인프라 스트럭처 코드화(IaC) 와 실제 서버 격리 모니터링 방식의 운영 마인드셋을 점검합니다.

- **꼬리 질문 예시**
  1. DB 볼륨 데이터를 매핑하지 않으면 재시작 시 어떤 치명적인 곤경에 빠질 수 있습니까?
  2. 컨테이너 환경의 포트 포워딩 간상에 충돌이 올 때 env를 거치는 유동 바인딩 방법의 팁은?
- **평가 기준**
  오퍼레이터(DevOps) 단에서의 융통성과 컨테이너 생태계의 파편화 방지 솔루션 경험치를 파악합니다.

## Q24. [심화] GitHub Actions CI/CD 파이프라인의 구성은 구체적으로 어떻게 되는가?
- **핵심 답변 포인트**
  - 지속적 통합 단위 테스팅 및 Lint 방어 구간
  - 패키지 의존성 / Gradle / Npm 캐싱의 빌드 최적화
  - 배포 파이프라인 민감 시크릿 정보 연동
- **모범 답변**
  수동 배포 및 휴먼 에러의 발생을 영구히 끊어내기 위해, GitHub Actions 파이프라인 CI/CD 도구를 연동했습니다. PR 신청 시 백엔드 JUnit 5 테스트와 프론트엔드의 ESLint 및 TSC 빌드가 의무적으로 검증되어 통과하지 않으면 Merging을 1차 잠금 하는 제약(Rules)을 두었습니다. Main에 흡수된 직후, `deploy.yml` 로 전개되는 서버 배포에선 매번 반복적으로 다운받는 패키지 종속성 캐시(`actions/cache`)를 엮어 빌드 소요 시각을 1/3로 극적으로 감소시켰습니다. AWS나 DB 비밀번호, SSH 암호는 레포지토리에 방치하지 않고 Github Secrets 보안 변수로 환경 변수에 주입시켜 완벽한 CD 자동 방파제를 지었습니다.
- **꼬리 질문 예시**
  1. 롤링 배포, 블루/그린 배포 기술 중 현재 구조에 도입한 다운타임 회피 전략은 무엇입니까?
  2. 서버 프로퍼티의 Secret yml 설정값을 주입할 때 배포 스크립트단에서의 생성 과정은 구체적으로 어떻게 삽입했습니까?
- **평가 기준**
  코드 형상 관리 전략부터 안전 지향적인 클라우드 배포 CI/CD 파이프라인의 총괄 운영 기법을 묻습니다.
