# 카테고리 1: 프론트엔드 아키텍처

> [!TIP] 면접 TIP
> 프론트엔드 면접에서는 **상태 관리의 기준(서버 vs 클라이언트)**과 **렌더링 최적화**가 중요합니다. 명확한 기준(Trade-off)을 제시하며 도구를 선택한 이유를 논리적으로 방어하세요.

## Q1. [기초] React 19의 주요 변경점과 DayPoo에서 활용한 기능을 설명해주세요.
- **핵심 답변 포인트**
  - React Compiler 도입으로 인한 메모이제이션 훅 감소
  - 비동기 처리를 위한 `use()` 훅 및 Actions 개념
  - Server Components 구조 이해 및 활용
- **모범 답변**
  React 19의 핵심 강점은 성능 최적화의 자동화입니다. React Compiler 도입 덕분에 `useMemo`나 `useCallback`과 같은 수동 처리 없이도 보일러플레이트 코드를 줄이고 렌더링 성능을 확보할 수 있었습니다. 특히 DayPoo에서는 잦은 알림과 데이터 페칭 시 비동기 Promise에 `use()` 훅과 Form Actions 기능을 연계했습니다. 이를 통해 비동기 처리 간 Suspense의 우아한 로딩 전이를 구현할 수 있었습니다.
- **꼬리 질문 예시**
  1. 수동 메모이제이션이 여전히 필요한 엣지 케이스는 무엇일까요?
  2. Action을 통해 낙관적 업데이트 로직을 어떻게 간소화할 수 있나요?
- **평가 기준**
  지원자가 React 생태계의 선언적 패러다임을 이해하고 최신 문법(RSC, Compiler)에 대응할 수 있는지 평가합니다.

## Q2. [중급] Redux Toolkit과 RTK Query를 함께 사용한 이유와 역할 분담 기준은?
- **핵심 답변 포인트**
  - 클라이언트 상태와 서버 상태의 독립
  - API 자동 캐싱, 중복 요청 방지, 캐시 무효화 제어
  - Optimistic Update (낙관적 업데이트) 구현
- **모범 답변**
  기존 Redux에 모든 데이터를 담는 방식은 비동기 Thunk 코드가 비대해지는 문제가 있었습니다. DayPoo에서는 '서버 상태'와 '클라이언트 상태'를 분리하는 전략을 택했습니다. 랭킹 데이터나 백엔드 리뷰 페칭은 철저히 RTK Query로 넘겨 자체 캐싱과 폴링(Polling)의 효율성을 취했습니다. 반대로 UI 테마나 로컬 입력 폼은 순수 Redux Toolkit으로 분리했습니다. 특히 알림 읽음 처리 시 RTK Query의 Optimistic Update 기능을 활용해 체감 지연 시간을 제로로 만들었습니다.
- **꼬리 질문 예시**
  1. RTK Query와 React Query(TanStack Query)를 비교했을 때, RTK를 택한 이유는?
  2. 서버 상태를 관리할 때의 캐시 수명(TTL)은 어떻게 조절했습니까?
- **평가 기준**
  단순 상태 관리 도구의 사용을 넘어서 목적별 분류와 네트워크 부하 최적화에 대한 원칙을 소유하고 있는지 봅니다.

## Q3. [중급] Context 3개와 Redux Store의 역할 분리 기준을 설명해주세요.
- **핵심 답변 포인트**
  - 빈번한 리렌더링 통제 전략
  - 전역 상태 범위의 국소화 체계 (AuthContext, ThemeContext 등)
  - Provider 중첩 계층 고려
- **모범 답변**
  모든 상태를 Redux에 의존하지 않기 위해 갱신 빈도를 기준으로 분리했습니다. `AuthContext`나 모달 팝업 가시성과 같이 앱 전역에서 변경 빈도가 매우 낮은 핵심 인증/UI 요소들은 Context API로 가볍게 구현하여 상위에 배치했습니다. Context의 특성상 내부 값이 바뀌면 소비하는 모든 하위 컴포넌트가 리렌더링 되기 때문에, 빠르게 데이터 조작이 일어나는 전역 랭킹 스토어나 복잡한 액션 맵은 Redux에 격리함으로써 불필요한 렌더링 트리를 방어하고 렌더 트리의 성능을 끌어올렸습니다.
- **꼬리 질문 예시**
  1. Context 의존성을 최소화하는 Provider 트리 구조 설계 방법은?
  2. Context의 값을 쪼개(Split) 렌더링을 최적화하는 전략을 아시나요?
- **평가 기준**
  React 계층 구조 최적화 이론과 런타임 성능을 실질적으로 조율할 수 있는지 측정합니다.

## Q4. [심화] RTK Query의 캐시 무효화 전략을 실제 구현 사례로 설명해주세요.
- **핵심 답변 포인트**
  - `tagTypes`를 통한 데이터 그룹화
  - Get 메서드의 `providesTags`
  - Mutation의 사이드이펙트 `invalidatesTags`
- **모범 답변**
  성능 최적화와 데이터 정합성 보장을 위해 RTK Query의 Tagging 체계를 적극 활용했습니다. 예를 들어 알림 목록 조회 Endpoint에 `providesTags: ['Notification']`을 부여하여 응답 데이터를 캐시 메모리에 적재합니다. 이후 유저가 ‘모두 읽음’ Mutation 엔드포인트를 호출할 경우, 성공 응답 시 `invalidatesTags: ['Notification']` 속성이 발생하도록 엮었습니다. 프론트엔드는 추가적인 코드 없이 백그라운드에서 자동으로 알림 쿼리를 재호출해 최신 상태로 동기화하게 됩니다.
- **꼬리 질문 예시**
  1. 개별 ID 단위의 섬세한 무효화(Tagging ID)는 어떻게 처리했나요?
  2. 불필요한 네트워크 호출을 막기위한 조건부 invalidation 설계법은 무엇인가요?
- **평가 기준**
  상태 관리 라이브러리의 깊은 사상을 이해하고, 데이터 오염 없이 동기화 전략을 구축할 수 있는지 봅니다.

## Q5. [중급] Recharts로 랭킹 차트를 구현할 때 겪은 기술적 과제는?
- **핵심 답변 포인트**
  - `ResponsiveContainer` 크기와 반응형
  - 커스텀 툴팁(Tooltip)과 미려한 UI 도입
  - SVG DOM 렌더링으로 인한 다량 데이터 성능 이슈
- **모범 답변**
  랭킹 시스템 시각화를 위해 Recharts를 활용하며 반응형 뷰포트 확보와 UI 심미성을 신경 썼습니다. `ResponsiveContainer`를 부모 객체에 연동해 디바이스 환경 제약 없이 100% 비율을 갖도록 설계하였고, DayPoo 프로젝트의 특징인 프리미엄 글래스모피즘(유리 질감)을 차트에도 맞추기 위해 Tooltip 컴포넌트를 직접 오버라이드하여 디자인했습니다. 대량 랭킹 데이터 노출 시 DOM 과부하 현상이 발견되었는데, 렌더 데이터 페이징 개수를 제한하고 복잡한 SVG 필터를 비활성화하는 최적화를 통해 렌더링(Lag)를 줄였습니다.
- **꼬리 질문 예시**
  1. Window 리사이즈 이벤트 시 차트 리렌더링 비용을 कैसे 최적화했나요?
  2. D3.js가 아닌 Recharts를 선택한 가장 큰 트레이드오프는 무엇일까요?
- **평가 기준**
  라이브러리 적용력과 컴포넌트 커스텀/튜닝 수준 및 프론트엔드 최적화 분석 역량을 검증합니다.

## Q6. [심화] Framer Motion의 layout 애니메이션을 랭킹 순위 변동에 적용한 방식은?
- **핵심 답변 포인트**
  - `layout` 속성을 통한 부드러운 위치 마이그레이션
  - `key` 렌더링 전략을 활용한 Reorder 인지
  - UI 물리 속성(Spring, config) 튜닝
- **모범 답변**
  단순한 재렌더링 시 깜빡이게 되는 리스트 UI를, 실시간으로 변동하는 화려한 랭킹 시스템처럼 보이도록 Framer Motion 엔진을 가져왔습니다. 각 아이템에 React 고유 `key` (userId)를 매핑하고 `<motion.li layout>` 속성을 할당했습니다. 이 방식으로 데이터 정렬(Sort) 순서가 바뀌더라도 DOM 리플로우(Reflow)를 가속 GPU 엔진이 감지해 자연스러운 Transform 이동 경로 애니메이션으로 변환해 냅니다. CSS Transition 대비 Spring 물리 엔진 값을 주입해 텐션감을 부여함으로써 극적인 몰입도를 달성했습니다.
- **꼬리 질문 예시**
  1. `AnimatePresence`는 리스트 삭제 애니메이션에 주로 어떻게 쓰이나요?
  2. 해당 라이브러리로 인한 초기 번들(Bundle) 용량 증대 문제는 어떻게 완화했습니까?
- **평가 기준**
  시각 디자인 컴포넌트에 대한 높은 완성도와 브라우저 렌더링(Reflow/Repaint) 차이를 인지하는지를 살핍니다.

## Q7. [중급] 관리자 페이지의 DataTable 컴포넌트를 범용적으로 설계한 방법은?
- **핵심 답변 포인트**
  - TypeScript의 `Generic<T>` 활용
  - Headless UI 패턴과 `ColumnDef` 배열 추상화
  - 페이지네이션 및 필터 상태 모듈 내장
- **모범 답변**
  관리자 대시보드 내 다수 포진된 유저, 랭킹, 신고 관리 패널들이 동일한 UI 코드를 중복 생산하는 것을 막고자 단일 `DataTable<T>` 범용 컴포넌트를 설계했습니다. 컬럼 헤더 이름과 데이터 매핑 구조를 외부에 위임(Headless 방식)하기 위해 `ColumnDef`를 Props로 삽입 받게 했습니다. 결과적으로 표의 껍데기와 정렬/페이징 기능, CSS 그리드 포맷은 `DataTable` 내부가 일임하고, 데이터 로직은 호출하는 부모가 책임지면서 관리자 패널의 개발 생산성을 월등히 높였습니다.
- **꼬리 질문 예시**
  1. Headless UI 아키텍처가 프론트엔드 생태계에서 중요해지는 이유는 무엇이라고 생각합니까?
  2. 중첩 객체를 표현해야 하는 특수한 Row 레코드는 어떻게 제어했나요?
- **평가 기준**
  단순 렌더링이 아닌 디자인 시스템(Design System) 구축 관점에서 인터페이스를 설계하는 능력을 봅니다.

## Q8. [심화] JWT 토큰 갱신 시 동시 요청 Race Condition을 어떻게 해결했나요?
- **핵심 답변 포인트**
  - Axios Interceptor 체이닝 구조 파악
  - `isRefreshing` 지시자 및 비동기 Promise 큐
  - 무한 루프 회피 메커니즘
- **모범 답변**
  Dashboard나 랭킹 페이지 진입 시, 다중 API 통신과 토큰 만료(401 응답)가 겹치면 토큰 재발급(Refresh API) 요청이 폭주하는 심각한 동시성 꼬임(Race Condition) 문제가 발생했습니다. 이를 막기 위해 Axios response 인셉터 단에 `isRefreshing` 락 플래그 변수를 두었습니다. 첫 번째 만료 응답에만 재발급 API를 호출하고, 이어지는 다른 동시 요청들은 `Promise Queue` 배열에 쌓아두는 대기열(Holding) 패턴을 구현했습니다. 토큰 재발급이 성공하는 즉시 새 Authorization 헤더를 장착하여 큐에 대기 중이던 약속들을 일괄 `resolve` 함으로써 요청 손실과 서버 폭격을 방어할 수 있었습니다.
- **꼬리 질문 예시**
  1. 재발급 API마저 401 처리가 났을 경우 로직 복구는 어떻게 설계했나요?
  2. `localStorage`와 `httpOnly Cookie` 중 JWT 저장 방식을 보안적으로 어떻게 채택하는 것이 낫습니까? 
- **평가 기준**
  고급 클라이언트 비동기 통신 제어력 및 인증 세션 관리의 예외 처리가 안정적인지 평가합니다.
