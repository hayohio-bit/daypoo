## Q9. [기초] Spring Boot에서 Redis를 사용한 이유와 활용 범위를 설명해주세요.
- **핵심 답변 포인트**
  - 관계형 데이터베이스 부하 분산 스펙
  - 랭킹을 위한 Sorted Set 집계 최적화
  - 무거운 AI 텍스트 결과 캐싱 처리
- **모범 답변**
  PostgreSQL로 모든 데이터를 조회할 경우 디스크 I/O 병목이 발생하므로, 지연 응답에 민감한 서비스를 개선하기 위해 Redis 인메모리를 선택했습니다. 활용 범위는 크게 세 가지입니다. 첫째, 랭킹 시스템 내 정렬된 `Sorted Set` 구조로 실시간 순위를 빠르게 산정합니다. 둘째, 10초 이상의 지연 시간이 걸리는 GPT AI 리포트 반환 값을 한시적으로 캐싱(Caching)하여 유저 이탈과 비용을 낮췄습니다. 셋째, 분산 인스턴스 시 활용 가능한 글로벌 세션 및 토큰 블랙리스트를 관리하는 핵심 허브로 편제시켰습니다.
- **꼬리 질문 예시**
  1. Redis 캐시가 날아가는 휘발성(Eviction) 상황을 겪을 경우에 대한 방벽은?
  2. Cache Aside 패턴과 Write Through 패턴의 차이는?
- **평가 기준**
  Redis의 다양한 내장 자료구조와 캐싱 패턴에 대한 이해 및 메모리 관리 역량을 확인합니다.

## Q10. [중급] Redis Sorted Set으로 랭킹을 구현한 구체적인 방법은?
- **핵심 답변 포인트**
  - 명령셋: ZINCRBY, ZREVRANGE, ZREVRANK
  - Prefix 기반의 구조화된 Key 설계 (도메인 분리)
  - TTL (로테이션 관리)
- **모범 답변**
  정렬에 최적화된 트리구조의 O(log(N)) 시간 복잡도를 띄는 Redis `Sorted Set`을 활용했습니다. 랭킹 기간이 나뉘어 있는 만큼 Redis 키는 `ranking:regional:2026-w14` 등 도메인/주간 별로 네이밍을 쪼개었습니다. 로직 흐름 상 배변 기능 사용 시 즉각적으로 `ZINCRBY`를 호출해 점수를 올립니다. 리스트 페이징에는 데이터의 일부만 당겨오는 `ZREVRANGE`를, 접속 유저의 1인 순위 호출에는 `ZREVRANK` 명령을 활용합니다. 무분별한 메모리 낭비를 줄이기 위해 Spring Scheduler가 월 단위가 끝난 키에 만료시간(TTL) 적용을 트리거 하도록 분리했습니다.
- **꼬리 질문 예시**
  1. `ZREVRANGE`의 Offset 성능과 최신 스펙 `ZSCAN` 간 비교 결과는?
  2. Redis 서버 다운타임 간 점수 누락을 방지하는 백업 수단은?
- **평가 기준**
  인메모리 DB의 고수준 명령어 활용 능력과, 무한 적재를 막는 스케일링/비용 절감 센스를 봅니다.

## Q11. [심화] 랭킹 동점 처리를 어떻게 해결했나요?
- **핵심 답변 포인트**
  - Redis 기본 동작의 사전순(Lexicographical) 한계 타파
  - Score 숫자 인코딩 기법 적용 (점수 + 타임스탬프)
  - 선착순 우선 정렬 고안
- **모범 답변**
  Redis는 기본적으로 점수(Score)가 같으면 문자(Member ID)의 사전순 배열로 랭킹을 가르므로, '먼저 점수를 얻은 유저'가 우위를 점하지 못하는 공정성 문제가 있었습니다. 이를 해결하기 위해 Double 치수로 기재되는 Score에 독자적인 인코딩을 적용했습니다. 정수부에는 본체의 랭킹 점수를 넣고, 소수부에는 `미래 설정 연도 Timestamp - 현재 액션 Timestamp` 를 삽입했습니다. 이로인해 가장 먼저 액션을 한 사람의 소수점이 더 커지게 되고, 단 한 번의 ZADD만으로도 '점수가 동일하면 선착순 우대'라는 보조 정렬 로직이 완벽하게 해결되어 랭크 공정성을 확보했습니다.
- **꼬리 질문 예시**
  1. Timestamp 역산을 위해 64비트 실수 자릿수 한계가 깨지는 부동소수점 오차는 없었나요?
  2. 인코딩된 Score를 사용자에게 반환 시 순수 점수 파싱은 어떻게 복원했나요?
- **평가 기준**
  문제가 닥쳤을 때 한계 상황을 DB 구조나 수리적 논리 안에서 기발하게 고도화 처리할 수 있는 엔진 개발 역량.

## Q12. [중급] 이벤트 기반 아키텍처로 알림을 구현한 이유와 방법은?
- **핵심 답변 포인트**
  - 핵심 도메인과 시스템의 느슨한 결합 (Decoupling)
  - 트랜잭션 책임 방어 로직 제거
  - @Async 스레드 풀 이용
- **모범 답변**
  기존 방식은 "기록 저장 로직" 속에 "알림 전송 메서드"를 직접(DI) 호출해야 했는데, 알림 통신에서 에러나 딜레이가 나면, 본 로직인 저장 서비스에도 지연 타격을 입는 강결합이 존재했습니다. 모듈 확장을 위해 Spring EDA를 도입했습니다. 랭킹 시스템 및 리뷰 작성 클래스는 단순히 `PooRecordCreatedEvent`나 `RankingChangedEvent` 이벤트만 발간(Publish)합니다. 외부 `NotificationEventListener` 클래스에서만 이를 구독(Listener)하게 함으로써 코어 코드를 비워냈습니다. 또한 발행된 이벤트를 처리할 시 `@Async`를 통해 새로운 백그라운드 스레드로 위임했습니다. 
- **꼬리 질문 예시**
  1. 메시징 큐(RabbitMQ 등)나 Kafka 대비 그냥 Spring ApplicationEvent를 쓴 이유는 무엇입니까?
  2. `@Async` 스레드 풀 환경 설정 및 ThreadPoolTaskExecutor 거부를 겪은 적이 있나요?
- **평가 기준**
  아키텍처 확장 시 결합도를 낮추는 객체지향 기반 클린 코드 설계/리팩토링 능력을 점검합니다.

## Q13. [심화] @TransactionalEventListener의 phase = AFTER_COMMIT을 사용한 이유는?
- **핵심 답변 포인트**
  - 롤백 방지 및 팬텀 이벤트(Phantom Event) 전파 방지
  - 데이터 정합성 방어 
- **모범 답변**
  일반적인 `@EventListener`는 기존 데이터베이스 트랜잭션 수명 내에 편입됩니다. 따라서 중간에 알림은 정상적으로 발송되었는데 나중에 SQL 로직 중 에러가 발생해 롤백되면, 존재하지 않는 팩트를 유저에게 쏘는 '유령 알림 현상(Phantom Event)'을 낳게 됩니다. 저는 이벤트 구독기에 옵션을 주어 `phase = TransactionPhase.AFTER_COMMIT`을 명시했습니다. 즉 비즈니스 원천 로직이 DB에 완벽하게 커밋되어 상태가 영구 보존된 직후에만 알림 스레드가 가동되게끔 설계해 데이터 정합성 결함을 철저히 차단했습니다.
- **꼬리 질문 예시**
  1. AFTER_COMMIT 상태에서의 발송 코드가 실패하면 원본 작업에 영향이 없도록 설계되었나요?
  2. 확실한 알림 전송을 보장하는 Transactional Outbox Pattern에 대해 들어보셨나요?
- **평가 기준**
  트랜잭션(ACID) 생명 주기에 대한 탁월한 이해도와 예외 발생 상황 통제 능력을 검증합니다.

## Q14. [중급] 관리자 API 검색/필터를 JPA Specification으로 구현한 이유와 방법은?
- **핵심 답변 포인트**
  - 동적 비즈니스 조건 (Dynamic Predicate) 
  - Criteria Builder 조립 및 코드 재사용
  - QueryDSL 도입 대체 방어
- **모범 답변**
  관리자 대시보드 API는 '날짜', '역할군', '검색어' 등 상황에 따라 옵션이 달라지는 N가지 경우의 다중 쿼리가 수시로 발동됩니다. 모든 조건마다 `@Query` 메서드를 만드는 것은 유지보수 재앙이라 판단하여, 코드 단에서 쿼리를 선언형으로 조립하는 JPA `Specification` 패턴을 택했습니다. 각각의 필터 조각을 반환하는 메서드들을 선언하고, 인입되는 파라미터가 Non-Null일 경우에만 `.and()` 문으로 이어 붙여 동적 `Predicate`를 구성하게 했습니다. 이로써 외부 라이브러리인 QueryDSL 도입 없이 순수 JPA 프레임워크 기술만으로도 유연한 동적 검색 체계를 만들어냈습니다.
- **꼬리 질문 예시**
  1. QueryDSL 대비 JPA Criteria API 체제가 가독성에 한계가 있는 부분을 어떻게 보완했나요?
  2. N+1 문제를 피하기 위해 Specification 안에서 fetch join은 어떻게 적용했나요?
- **평가 기준**
  다양한 ORM 조건 필터링에 대한 체계적 설계와 중복 쿼리 제거 최적화를 검증합니다.

## Q15. [심화] AOP로 Rate Limiting과 관리자 감사 로그를 구현한 부분 설명해주세요
- **핵심 답변 포인트**
  - 관심사 분리(AOP)의 가치
  - 커스텀 어노테이션 기반 타겟팅 및 @Around
  - MDC 기반 식별자 삽입 및 Redis 카운팅 매핑
- **모범 답변**
  관리자 백오피스의 주요 통신마다 1초 트래픽 제한(초당 N회 한계)과 '누가 조작했는지'를 알아내는 Audit log가 필요했습니다. 하지만 각 비즈니스 모델(컨트롤러) 속에 해당 방어 코드를 욱여넣으면 객체 독립성이 무너집니다. 따라서 Spring AOP(Aspect Oriented Programming)를 차용해, 방어 대상 컨트롤러에만 달리는 `@AdminAuditLog` 커스텀 어노테이션을 만들었습니다. `@Around` 로직을 통해 실행 직전 Redis에 도달 횟수 카운터를 비교 차단하고, 실행에 성공하면 조작 액션과 이메일을 파싱해 강제 로깅했습니다. 결과적으로 서비스 코드는 수정 없이 애너테이션 한 줄 부여만으로 핵심 인프라 보안 능력을 탑재하게 되었습니다.
- **꼬리 질문 예시**
  1. AOP Proxy가 걸렸을 때 내부 함수 간 호출 시 인터셉터가 작동하지 않는 이유는?
  2. 다중 서버의 Rate Limit 통제를 위해 분산 Lock 알고리즘은 접목했나요?
- **평가 기준**
  객체 결합도를 낮추는 Spring 인프라 아키텍처에 대한 통달력과 보안 지표 활용을 살핍니다.

## Q16. [중급] Flyway 마이그레이션 도구 도입 배경 및 전략
- **핵심 답변 포인트**
  - 무중단 연속 배포 시 스키마 동기화 충돌 이슈
  - `V버전__이름.sql` 파일 관리 시스템 및 BaseLine 지정방식
  - JPA ddl-auto 무력화
- **모범 답변**
  초반엔 Spring JPA의 `ddl-auto: update`에 의존했으나, 여러 협업 환경과 서버 재배포로 인해 버전 충돌이나 컬럼 분실, 데이터 유실의 심각성에 노출되었습니다. 이를 방어하기 위해 DB 스키마 형상 툴인 Flyway를 도입했습니다. 엔티티 수정에 닿기 전 V1부터 V23까지 `V{Version}__{Description}.sql` 룰을 엄격하게 지켜 스크립트를 누적해나갔고, DB 서버에는 영구히 수정 기록만 `history` 테이블에 남겨 검증되게 했습니다. 개발 서버 간 DB 상태가 불일치할 시 충돌을 뿜어 빌드 자체를 사전 차단하므로 데이터 무결성과 팀 내 버그 예방 안전망 역할을 확립했습니다.
- **꼬리 질문 예시**
  1. 데이터가 들어가 있는 운영 DB 컬럼의 타입을 변경해야 될 때의 롤백/복구 전략은 있나요?
  2. JPA Entity의 필드명과 DB 컬럼의 Sync 단락이 어긋날 땐 어떤 컨벤션/테스트로 방어하나요?
- **평가 기준**
  운영 및 배포 시 가장 골치 아픈 데이터 스키마 관리에 관한 성숙한 파이프라인 관리 능력을 체크합니다.
