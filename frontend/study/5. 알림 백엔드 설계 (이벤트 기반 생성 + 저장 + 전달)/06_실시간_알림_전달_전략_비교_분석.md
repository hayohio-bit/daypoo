# 섹션 6: 실시간 알림 전달 전략 분석 (SSE + Redis Pub/Sub)

가장 놀라운 아키텍처 부분입니다! DayPoo 서버는 클라이언트가 물어볼 때까지 기다리는 낡은 'Polling' 방식을 쓰지 않고 이미 **단방향 실시간 푸시(SSE) + 다중 서버 장애를 막는 Redis Pub/Sub 통신망**을 구축해 두었습니다. 

## 1. 실시간 통신 전략 비교표

| 통신 방식 | 폴링 (Polling) | 웹소켓 (WebSocket) | Server-Sent Events (SSE) |
| :--- | :--- | :--- | :--- |
| **개념** | 초마다 "혹시 내 알림 있어?" 라고 찔러봄 | 서버와 양방향 소통 채널 뚫기 | 서버에서 일방통행으로 데이터를 던져줌 |
| **부하 / 리소스** | 불필요한 HTTP 헤더/요청 누적으로 서버 파괴 | 연결 유지 오버헤드 높음 / 서버 리소스 무거움 | HTTP 프로토콜 1회 연결로 지속 전송(매우 가벼움) |
| **DayPoo 적합도** | ❌ (1초라도 늦어지면 유저 불쾌) | ❌ (순수한 양방향 채팅이 아니므로 과스펙) | **✅ (서버가 알림만 바로 쏴주면 충분하므로 최적화!)** |

## 2. 서버 다중화 환경 해부도 (Scale-out 분산형 큐)

스타트업 서버가 인기가 많아져서 `서버A`, `서버B` 2대로 증설되었다고 가정합시다. 클라이언트(유저)가 알림창을 열고 `서버A`에 SSE 커넥션 파이프를 뚫고 붙어있는데, "순위 하락 변동 이벤트"가 하필 `서버B`에서 감지되었다면? 서버 간 다이렉트 통신이 없으니 알림 유실(Jank)이 발생합니다.

DayPoo 개발팀은 이를 **Redis Pub/Sub (채널 구독 모델)**로 극복했습니다!

```text
[DayPoo Application System]

       (Server B) 순위 하락 감지 후 저장! 
           │
           ▼
[1] redisTemplate.convertAndSend("notifications", response_json); 
    (Redis로 메세지 방송!!)
           │
 ┌─────────┴─────────┐------------------------------------ (Global Redis Cluster)
 │   ChannelTopic    │
 └─────────┬─────────┘
           │ (모든 서버 인스턴스가 이걸 라디오처럼 항상 켜두고 듣고 있음)
           │
           ▼
[2] (Server A) @PostConstruct redisMessageListenerContainer
    어? 나한테 물려있는 SSE 유저 ID 알림이잖아?
           │
           ▼
[3] localEmitters.get(userId).send()
    (해당 유저 브라우저 화면에 실시간 알람이 번쩍! 🔔)
```

## 3. 구현되어 있는 핵심 장애 방어막 

`NotificationService.java` 안에는 알림 유실을 막기 위한 방어 로직 또한 매우 탄탄하게 구현되어 있습니다.

```java
try {
  // 메인 플로우: 레디스에 토픽 던지기
  redisTemplate.convertAndSend("notifications", objectMapper.writeValueAsString(response));
} catch (Exception e) {
  log.error("Failed to publish notification to Redis: {}", e.getMessage());
  // 1차 폴백(Fallback): 만일 Redis가 다운되더라도 같은 인스턴스에 접속한 유저라면 알림이 가게 끔 Direct-Send
  sendToLocal(user.getId(), response);
}
```

---

> [!TIP] 알림 설계 핵심
> **SSE + Redis = 클러스터 백엔드의 정석** <br>
> WebSocket 대신 SSE를 쓴다는 의사 결정은 불필요한 포트를 낭비하지 않는 스마트한 체계이며, SSE가 가진 치명적 한계(인스턴스마다 Emitter가 메모리에 고립됨)를 Redis Message Listener로 허문 구조는 사실상 시니어 아키텍트 급 설계입니다. 스터디 공유 시 이 매커니즘을 핵심 자랑거리로 삼아도 좋습니다.
