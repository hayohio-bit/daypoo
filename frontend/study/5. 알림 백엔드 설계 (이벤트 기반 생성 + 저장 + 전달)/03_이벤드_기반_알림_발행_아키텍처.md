# 섹션 3: 이벤트 기반 알림 발행 아키텍처 (EDA)

**[정합성 검증 완료 사항]**
> 현재 프로젝트의 `event/` 폴더에는 `PooRecordCreatedEvent`, `ToiletReviewCreatedEvent` 2가지만 존재하며, 랭킹 변동을 감지하는 `RankingChangedEvent`는 **현재 미구현 상태**입니다. 아래 내용은 향후 알림 발행 흐름 도입을 위한 설계 제안입니다.

객체 지향 시스템에서 "랭킹 서비스"가 "알림 전송 로직"을 직접 호출하면 결합도가 상승하고 지연 시간이 증가합니다. 이를 분리하는 **이벤트 구동 설계(Event-Driven Architecture)**를 분석합니다.

## 1. 이벤트 발행-구독 플로우 다이어그램 (Sequence)

```text
[RankingService]                        [ApplicationEventPublisher]             [NotificationEventListener]
       │                                            │                                      │
       │ 1. 랭킹 스코어 업데이트 (DB 반영)          │                                      │
       ├──────────────────────────────────────────> │                                      │
       │                                            │                                      │
       │ 2. RankingChangedEvent 객체 생성 및 발행   │                                      │
       ├──────────────────────────────────────────> │                                      │
       │                                            │                                      │
       │ 3. 트랜잭션 정상 커밋 대기                 │                                      │
       │    (Ranking 데이터 확정)                   │                                      │
       │............................................├── 4. @TransactionalEventListener ───>│
       │                                            │    (phase = AFTER_COMMIT)            │
       │                                            │                                      │
                                                                                           │ 5. @Async 스레드로 위임
                                                                                           ├──────────────────────> [NotificationService]
                                                                                           │                          │ 6. DB 알림 저장
                                                                                           │                          │ 7. Redis Pub/Sub 발행 
```

## 2. [스터디용 제안] 커스텀 이벤트 및 리스너 구현

현재 DayPoo 프로젝트의 이벤트 폴더 구조 형태에 맞춰 적용할 수 있는 스프링 부트 이벤트 발행/구독 코드 세트입니다.

### 2-1. 이벤트 객체 (POJO)
이벤트 객체는 단순한 상태 전달자(Data Carrier) 역할만 수행합니다.

```java
// RankingChangedEvent.java
package com.daypoo.api.event;

import com.daypoo.api.entity.User;
import lombok.AllArgsConstructor;
import lombok.Getter;

@Getter
@AllArgsConstructor
public class RankingChangedEvent {
    private final User user;
    private final int oldRank;
    private final int newRank;
}
```

### 2-2. 이벤트 리스너 (Spring Context)
스프링의 트랜잭션 생명주기를 감지하는 리스너입니다. 알림 때문에 메인 비즈니스가 롤백되지 않도록 격리합니다.

```java
// NotificationEventListener.java
package com.daypoo.api.event;

import com.daypoo.api.entity.enums.NotificationType;
import com.daypoo.api.service.NotificationService;
import lombok.RequiredArgsConstructor;
import org.springframework.scheduling.annotation.Async;
import org.springframework.stereotype.Component;
import org.springframework.transaction.event.TransactionPhase;
import org.springframework.transaction.event.TransactionalEventListener;

@Component
@RequiredArgsConstructor
public class NotificationEventListener {

    private final NotificationService notificationService;

    /**
     * 메인 트랜잭션이 성공적으로 COMMIT 된 이후에만 비동기로 실행됩니다.
     */
    @Async
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void handleRankingChangedEvent(RankingChangedEvent event) {
        String title = "순위 변경 알림";
        String content = String.format("회원님의 순위가 %d위에서 %d위로 변동되었습니다.", 
                                       event.getOldRank(), event.getNewRank());
                                       
        // NotificationService (DB 저장 및 SSE 전송) 위임
        notificationService.send(
            event.getUser(), 
            NotificationType.SOCIAL, 
            title, 
            content, 
            "/ranking"
        );
    }
}
```

---

> [!TIP] 알림 설계 핵심
> **데이터 정합성과 트랜잭션 분리 지연** <br>
> `@EventListener`를 사용하면 이벤트가 원래 트랜잭션에 강제로 묶입니다. 만약 알림 전송 중 오류가 나면 '랭킹 변동' 내역 전체가 롤백(Rollback)되는 치명적 사고가 납니다. `@TransactionalEventListener(phase = AFTER_COMMIT)`을 활용하면 메인 로직(DB Update)이 안전하게 확정된 후 스레드를 분기 시켜 독립성을 확보할 수 있습니다.

> [!WARNING] 성능 고려 사항
> **이벤트 실패 재시도 로직 (Fallback)** <br>
> `@Async` 스레드에서 알림 생성이 실패하더라도 사용자는 오류를 알 수 없습니다. 치명적인 결제/구독 알림의 경우 `Spring Retry` 어노테이션이나 발행 실패 내역을 로깅(Dead Letter Queue 형태)하는 구조를 추가해야 안정성이 보장됩니다.
