# 섹션 7: 관리자 알림 발송 및 스케줄러 고려 사항

일반적인 랭킹 및 건강 리포트 알림 외에도, 운영진이 개입하는 강제 푸시 알림과 사용자 개입이 없는 시간 스케줄 트리거(Crontab)에 대해 제안합니다. (현재 구현 외 확장 스펙)

## 1. 어드민(Admin) 애플리케이션 공지 시스템

시스템 점검, 긴급 롤백, 전체 보상(이벤트)을 진행할 때는 시스템(System) 차원의 일괄 공지가 필요합니다. 기존의 `NotificationService`에서는 `NotificationRepository.saveAll(List<Notification>)`을 사용하여 배치(Batch) 저장을 수행할 수 있어야 합니다.

```java
// 제안 로직: 관리자 다중 메시징 기능 (가칭 AdminManagementService 내부)

@Transactional
public void sendSystemAnnouncementToAll(String title, String content) {
    // 주의: 실제 상용에선 수만 명 스레드가 멈추므로 Spring Batch를 사용하여 청크(Chunk) 단위로 쪼개야 함
    List<User> activeUsers = userRepository.findAllByIsActiveTrue();
    
    List<Notification> announcements = activeUsers.stream()
        .map(user -> Notification.builder()
            .user(user)
            .type(NotificationType.SYSTEM)
            .title(title)
            .content(content)
            .redirectUrl("/notices") // 공지사항 게시판 라우팅
            .build()
        ).toList();
        
    // Bulk Insert 최적화
    notificationRepository.saveAll(announcements);
    
    // (선택) 접속해있는 유저에 한해 SSE 실시간 알림 방출
    // ...
}
```

## 2. 알림 자동화 보존 TTL (스케줄러/Job)

데이터베이스가 수평 및 수직 확장되더라도 찌꺼기 데이터(읽음 처리된 6개월 전 알림 등)를 방치하면 I/O 성능이 급격히 저하됩니다.

- **스프링 자체 크론(Cron):** 가장 단순하며 @Scheduled 어노테이션으로 매일 새벽 4시에 삭제 트리거를 만듭니다. (단일 서버에 유리)
- **Redis 만료 시간 (TTL):** SSE 스트림 전송의 속도를 위해 DB를 안 거치고 In-Memory 캐시로 알림을 서빙할 때 자동 만료를 사용합니다.

```java
// 제안 로직: 글로벌 스케줄링 파일 작성 시
@Component
@RequiredArgsConstructor
public class NotificationCleanupScheduler {

    private final NotificationRepository notificationRepository;

    /**
     * 초 분 시 일 월 요일 - 매일 새벽 4시에 작동
     */
    @Scheduled(cron = "0 0 4 * * ?")
    @Transactional
    public void cleanupOldNotifications() {
        // 생성일이 현재 시간부터 30일(1개월) 전인 알림 데이터를 전부 색출
        LocalDateTime thirtyDaysAgo = LocalDateTime.now().minusDays(30);
        
        // 성능 향상을 위해 JPA 더티체킹이 아닌 커스텀 JPQL 로 직접 삭제
        // notificationRepository.deleteByCreatedAtBefore(thirtyDaysAgo);
    }
}
```

---

> [!WARNING] 성능 고려 사항
> **Bulk Insert 설정 체크** <br>
> `saveAll`을 사용하면 1만 명에게 공지를 보낼 때 여전히 내부적으로 `INSERT INTO` 쿼리가 무려 10,000번 발사됩니다. `application.yml` 단말에 `spring.jpa.properties.hibernate.jdbc.batch_size=500`과 `order_inserts=true` 설정을 활성화하여 JPA가 인서트 구문을 한 문장으로 합치도록(Batched Insert) 유도하는 것이 성능의 생명줄입니다.
