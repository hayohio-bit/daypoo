## 섹션 1: main.tsx 엔트리포인트 분석

**[기초]**

### 1.1 실제 코드

```tsx
// main.tsx
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import './index.css'
import App from './App'

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <App />
  </StrictMode>,
)
```

main.tsx는 **극도로 심플**하다. React 19의 `createRoot` API로 `#root`에 마운트하며, 모든 Provider 래핑은 `App.tsx` 내부에서 처리한다.

### 1.2 Provider 래핑 순서 (App.tsx 내부)

```
StrictMode
└── BrowserRouter          ← ① 라우팅 (최외곽: navigate() 등 모든 곳에서 필요)
    └── LazyMotion          ← ② 애니메이션 엔진 (페이지 전환에 필요)
        └── ErrorBoundary   ← ③ 에러 캐치 (하위 모든 컴포넌트의 에러 잡기)
            └── AuthProvider ← ④ 인증 상태 (API 호출 전 토큰 관리)
                └── TransitionProvider ← ⑤ 페이지 전환 커튼 (navigate 의존)
                    └── NotificationProvider ← ⑥ 알림 (Auth 의존: 로그인 상태 확인)
                        └── Suspense + Routes
```

### 1.3 래핑 순서의 설계 근거

| 순서 | Provider | 의존 대상 | 이유 |
|------|----------|-----------|------|
| ① | `BrowserRouter` | 없음 | `useNavigate()`, `useLocation()`의 근원. 모든 하위에서 필요 |
| ② | `LazyMotion` | 없음 | `lazy()` 페이지 전환 시 애니메이션 기능 제공. `domAnimation` 동적 로드 |
| ③ | `ErrorBoundary` | 없음 | 하위 전체의 렌더링 에러를 잡는 최상위 안전망 |
| ④ | `AuthProvider` | API 클라이언트 | 토큰 관리, 로그인/로그아웃. NotificationProvider보다 먼저 필요 |
| ⑤ | `TransitionProvider` | `useNavigate()` | 페이지 전환 시 페인트 커튼 애니메이션. Router 의존 |
| ⑥ | `NotificationProvider` | `useAuth()` | 로그아웃 시 알림 초기화. Auth에 직접 의존 |

### 1.4 Context vs Redux 사용 분기 기준

> **핵심 발견:** DayPoo는 **Redux를 사용하지 않는다.**
> `node_modules` 분석에서 Redux가 나타난 것은 다른 라이브러리의 의존성(transitive dependency)이며,
> `package.json`과 소스코드 어디에도 Redux import가 없다.

| 기준 | Context API (DayPoo 선택) | Redux Toolkit |
|------|--------------------------|---------------|
| **상태 범위** | 인증, 알림, 페이지전환 (3개) | 대규모 글로벌 상태 |
| **업데이트 빈도** | 낮음 (로그인/로그아웃, 알림 수신) | 높음 (실시간 리스트 조작) |
| **보일러플레이트** | 최소 | slice, thunk, selector 필요 |
| **적합한 규모** | 3~5개 전역 상태 | 10개 이상 상호의존 상태 |
| **DayPoo의 판단** | 17개 페이지지만 전역 상태가 3개뿐 → Context로 충분 | 불필요한 복잡성 |

> **핵심 설계 포인트**
>
> DayPoo는 전역 상태를 3개의 Context로 관리하며, 페이지별 로컬 상태는 각 컴포넌트
> 내부의 `useState`/`useCallback`으로 처리한다. 데이터 페칭은 커스텀 훅
> (`useToilets`, `useRankings`)이 담당하며, API 클라이언트(`apiClient.ts`)는
> 싱글턴 클래스로 토큰 리프레시와 지수 백오프 재시도를 내장한다.

### 2.2 라우트 가드 패턴

DayPoo는 **ProtectedRoute를 사용하지 않는다.** 대신 두 가지 전략을 병용한다:

#### 전략 1: AdminRoute (명시적 가드)

```tsx
function AdminRoute({ children }: { children: React.ReactNode }) {
  const { user, loading } = useAuth();

  if (loading) return <LoadingScreen />;

  // role이 문자열일 수도, 배열일 수도 있는 방어적 코딩
  const isAdmin = user && (
    (typeof user.role === 'string' && user.role.toUpperCase().includes('ADMIN')) ||
    (Array.isArray(user.role) && user.role.some(r => r.toUpperCase().includes('ADMIN')))
  );

  if (!isAdmin) return <Navigate to="/main" replace />;
  return <>{children}</>;
}
```

#### 전략 2: API 레벨 401 + 자동 리프레시 (일반 사용자)

```
MyPage 접근 → API 호출 → 401 반환
                           ↓
                   apiClient.tryRefreshToken()
                           ↓
                   성공? → 재요청   실패? → 토큰 삭제 + 에러
```

> 일반 사용자 페이지(MyPage, PaymentSuccess)는 프론트 가드 없이 접근 가능하되,
> API 호출 시 401이 발생하면 `apiClient`가 자동으로 토큰 리프레시를 시도한다.

### 2.3 코드 스플리팅 전략

```tsx
// named export를 lazy로 처리하는 패턴
const MainPage = lazy(() =>
  import('./pages/MainPage').then(m => ({ default: m.MainPage }))

```

## 섹션 3: 상태 관리 아키텍처

**[중급]**

### 3.1 전체 상태 관리 구조

DayPoo는 **Context API + 커스텀 훅 + 싱글턴 API 클라이언트** 3계층으로 상태를 관리한다.

```
┌──────────────────────────────────────────────────┐
│                   Context 계층                     │
│  AuthContext ← NotificationContext                 │
│  TransitionContext (독립)                           │
├──────────────────────────────────────────────────┤
│               커스텀 훅 계층                        │
│  useToilets  useRankings  useGeoTracking          │
│  useCursorPosition  usePaintTransition            │
├──────────────────────────────────────────────────┤
│            API 클라이언트 계층                      │
│  ApiClient (싱글턴 클래스)                         │
│  ├─ 자동 토큰 리프레시 (뮤텍스 패턴)               │
│  ├─ 지수 백오프 재시도 (최대 3회)                   │
│  └─ AbortController 타임아웃 (30초)                │
└──────────────────────────────────────────────────┘
```

### 3.2 Context 3개의 역할과 상태

#### AuthContext — 인증 상태 관리

```tsx
interface AuthContextType {
  user: UserResponse | null;    // 현재 로그인 사용자 정보
  loading: boolean;             // 인증 확인 중 여부
  login: (accessToken, refreshToken, stayLoggedIn?) => Promise<void>;
  logout: () => Promise<void>;
  deleteMe: () => Promise<void>;
  refreshUser: () => Promise<void>;
  isAuthenticated: boolean;     // !!user 파생값
}
```

**토큰 저장 전략:**

| 옵션 | 저장소 | 만료 |
|------|--------|------|
| "로그인 유지" 체크 | `localStorage` | 3일 (수동 관리) |
| 체크 안 함 | `sessionStorage` | 브라우저 닫으면 삭제 |

#### NotificationContext — 알림 + 토스트

```tsx
interface NotificationContextType {
  notifications: Notification[];          // 서버 알림 목록
  unreadCount: number;                    // useMemo 파생값
  showToast: (title, message, type?, icon?) => void;  // 5초 자동 삭제
  fetchNotifications: () => Promise<void>;
  markAllAsRead: () => Promise<void>;
  markAsRead: (id) => Promise<void>;
  deleteNotification: (id) => Promise<void>;
  setNotifications: Dispatch<...>;        // 외부에서 직접 조작 가능
}
```

**의존성:** `useAuth()` 호출 → 로그아웃 시 알림 초기화

#### TransitionContext — 페이지 전환 애니메이션

```tsx
interface TransitionContextType {
  transitionTo: (path: string) => void;   // 커튼 내려오기 → navigate → 커튼 올라가기
  phase: 'down' | 'up' | 'idle';
}
```

**상태 흐름:**
```
idle → down (커튼 내려옴) → navigate() 실행 → up (커튼 올라감) → idle
```

### 3.3 API 클라이언트 싱글턴 패턴

```tsx
class ApiClient {
  // F1: AbortController 타임아웃 (30초)
  // F2: 지수 백오프 재시도 (1s → 2s → 4s, 최대 3회)
  // F3: 토큰 리프레시 뮤텍스 (동시 401 발생 시 1회만 리프레시)

  private refreshPromise: Promise<boolean> | null = null;  // 뮤텍스

  private async tryRefreshToken(): Promise<boolean> {
    if (this.refreshPromise) return this.refreshPromise;  // 이미 진행 중이면 재사용
    this.refreshPromise = this.doRefreshToken();
    try { return await this.refreshPromise; }
    finally { this.refreshPromise = null; }
  }
}

export const api = new ApiClient();  // 싱글턴 인스턴스
```

**재시도 제외 엔드포인트:** `/auth/login`, `/auth/signup`, `/auth/refresh`
**재시도 제외 상태 코드:** 400, 401, 403, 404, 422, 429

### 3.4 상태 저장 위치 분류

| 상태 종류 | 저장 위치 | 예시 |
|-----------|----------|------|
| 전역 인증 | AuthContext | user, isAuthenticated |
| 전역 알림 | NotificationContext | notifications[], unreadCount |
| 전역 전환 | TransitionContext | phase, transitionTo |
| 페이지 로컬 | 각 페이지 useState | 탭 상태, 모달 open, 선택된 아이템 |
| 서버 데이터 | 커스텀 훅 (fetch → useState) | toilets[], rankings |
| 영속 토큰 | localStorage / sessionStorage | accessToken, refreshToken |

> **핵심 설계 포인트**
>
> - Redux 없이 Context 3개 + 커스텀 훅으로 전체 상태 관리
> - API 클라이언트가 인프라 관심사(토큰 리프레시, 재시도, 타임아웃)를 캡슐화
> - `useMemo`로 Context value 메모이제이션 → 불필요한 리렌더 방지
> - 서버 상태는 훅 내부 useState에 캐시 (React Query/RTK Query 미사용)

## 섹션 4: 커스텀 훅 5개 분석

**[심화]**

### 4.1 훅 목록 및 역할

| 훅 | 역할 | 사용처 |
|----|------|--------|
| `useToilets` | 화장실 데이터 페칭 + 상태 관리 | MapPage, HeroSection, MapSection |
| `useGeoTracking` | GPS 추적 + 자동 체크인 | MapPage |
| `useRankings` | 랭킹 데이터 페칭 | RankingPage |
| `useCursorPosition` | 마우스 좌표 추적 + 모바일 감지 | CustomCursor |
| `usePaintTransition` | 페이지 전환 상태 관리 | (TransitionContext로 대체됨) |

### 4.2 useToilets — 화장실 데이터 훅

```tsx
function useToilets(options: {
  lat: number;
  lng: number;
  radius?: number;          // 기본 1000m
  bounds?: { swLat, swLng, neLat, neLng } | null;
  level?: number;           // 카카오맵 줌 레벨
  visitedIds?: Set<string>;
  favoriteIds?: Set<string>;
}): {
  toilets: ToiletData[];
  loading: boolean;
  error: string | null;
  toggleFavorite: (id: string) => void;
  markVisited: (id: string) => void;
  refetch: () => Promise<void>;
}
```

**내부 구현 핵심:**

```
1. bounds가 있으면 → 중심점 재계산 + 동적 반경 (Haversine 공식)
2. 줌 레벨에 따른 반경 제한: level >= 7 → 최대 5km, 아니면 3km
3. 결과 최대 1000개 제한 (성능 보호)
4. 300ms 디바운스 → bounds 변경 시 즉시 호출 방지
5. favoriteIds/visitedIds가 비동기 로드 → useEffect로 후속 동기화
6. Ref 패턴: visitedIdsRef, favoriteIdsRef → 클로저 stale 값 방지
```

### 4.3 useGeoTracking — GPS + 자동 체크인

```tsx
function useGeoTracking(
  toilets: ToiletData[],
  onAutoCheckIn?: (remainedSeconds: number) => void,
  isEnabled?: boolean       // 기본 true
): {
  position: { lat: number; lng: number } | null;
  granted: boolean;
}
```

**자동 체크인 (Fast Check-in) 로직:**

```
watchPosition() 콜백
  ↓
각 화장실과 거리 계산 (Haversine)
  ↓
150m 이내 진입 감지?
  ↓ Yes
마지막 체크인에서 120초 경과?
  ↓ Yes
POST /records/check-in 호출
  ↓
refreshUser() → onAutoCheckIn() 콜백
```

### 4.4 useRankings — 랭킹 데이터

```tsx
function useRankings(
  tab: 'total' | 'local' | 'health',
  regionName?: string
): {
  data: RankingResponse | null;
  loading: boolean;
  error: string | null;

}
```

**API 엔드포인트 매핑:**
- `total` → `GET /rankings/global`
- `local` → `GET /rankings/region?regionName=...`
- `health` → `GET /rankings/health`

**방어 로직:** `local` 탭인데 `regionName`이 없으면 API 호출하지 않음

### 4.5 useCursorPosition — 커스텀 커서

```tsx
function useCursorPosition(): {

}
```

모바일 감지 시 커스텀 커서 비활성화에 사용.

### 4.6 usePaintTransition — 페이지 전환 (레거시)

TransitionContext로 리팩터링되어 현재는 미사용. Context 버전과 동일한 `down → navigate → up → idle` 상태 머신.

> **핵심 설계 포인트**
>
> - `useToilets`가 가장 복잡: Haversine 동적 반경 + 디바운스 + Ref로 stale closure 방지
> - `useGeoTracking`은 `watchPosition`으로 실시간 위치 추적, 150m 진입 시 자동 체크인
> - 모든 데이터 훅이 `useState` + `useCallback` + `useEffect` 조합 (외부 라이브러리 없음)
> - React Query 등 없이 직접 구현 → 캐시/무효화 전략이 수동적 (refetch 명시 호출)

### 6.2 Framer Motion 패턴

#### LazyMotion + domAnimation (번들 최적화)

### 6.3 DiceBear 아바타 전략

**설치된 30여 개 스타일 중 실제 사용 5개:**

```tsx
import { funEmoji, avataaars, bottts, lorelei, pixelArt } from '@dicebear/collection';
```

| 스타일 | 용도 | 시각적 특징 |
|--------|------|------------|
| `funEmoji` | **기본 사용자 아바타** | 이모지 조합, 귀여움 |
| `avataaars` | 상점 AVATAR 아이템 | 픽사 스타일, 친근함 |
| `pixelArt` | 상점 EFFECT 아이템 | 레트로 픽셀, 독특함 |
| `bottts` | 기타 아이템 폴백 | 로봇 스타일 |
| `lorelei` | 예비 (코드에 import) | 만화 스타일 |

**시드 기반 생성 로직:**

```tsx
export const generateAvatar = (seed: string | number, style = 'funEmoji', size = 256) => {
  const avatar = createAvatar(AVATAR_STYLES[style], {
    seed: `daypoo-${seed}`,   // 'daypoo-' 접두사로 고유성 보장
    size,
  });
  return avatar.toDataUri();  // data:image/svg+xml;base64,...
};
```

**imageUrl 파싱 프로토콜:**

```
imageUrl 값 분기:
├── "dicebear:funEmoji:seed123"  → DiceBear 동적 생성
├── "https://..."                → 외부 URL 그대로 사용
├── "🎉" (이모지)                → 이모지 텍스트 표시
├── null/undefined               → generateItemAvatar() 폴백
```

```tsx
// 이모지 판별 유틸
export const isEmoji = (str: string | null | undefined): boolean => {
  if (!str) return false;
  return str.length <= 8 && /\p{Extended_Pictographic}/u.test(str);
};
```

**용도별 아바타 함수:**

| 함수 | 크기 | 용도 |
|------|------|------|
| `generateProfileAvatar()` | 256px | 마이페이지 프로필 |
| `generateRankingAvatar()` | 128px | 랭킹 목록 |
| `generateSmallAvatar()` | 48px | 댓글/리뷰 |
| `generateItemAvatar()` | 200px | 상점 아이템 미리보기 |

> **핵심 설계 포인트**
>
> - LazyMotion + domAnimation으로 Framer Motion 번들 최적화
> - `<m.div>` (경량) vs `<motion.div>` (풀기능) 구분 사용
> - DiceBear 30+개 설치되었지만 실사용 5개 → 나머지는 상점 확장용 예비
> - `dicebear:style:seed` 커스텀 프로토콜로 서버-클라이언트 아바타 동기화
> - PaintCurtain은 SVG path 기반 커스텀 와이프 효과 (CSS transition이 아닌 JS 계산)
